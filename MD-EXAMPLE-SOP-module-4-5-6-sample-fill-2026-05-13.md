# MD-EXAMPLE — SOP 模組 4～6 範例填寫版 — 2026-05-13

Status: draft
Purpose: 示範 SOP 模組 4～6（資料、環境、驗收）一份比較好的填法，讓一般使用者看得懂要準備什麼、在哪裡操作、以及怎樣才算真的完成。

Important:
- 這份是**示範用**，不是正式規格。
- 內容是用「假想專案管理 / 接案流程系統」當例子。
- 真正新專案時，應依實際需求重填。

---

# 模組 4：資料蒐集清單（範例填寫）

## 資料項目 1
- 資料名稱：系統名稱
- 這是什麼資料：這套系統未來對內對外要怎麼稱呼。
- 為什麼需要這筆資料：會影響首頁標題、文件命名、對外溝通與帳號設定。
- 是否必填：是。
- 如果缺少，會造成什麼影響：專案啟動時會一直用暫時名稱，後面容易混亂。
- 最晚要在哪個階段補齊：專案啟動前。
- 這筆資料會不會影響正式系統結構：通常不影響 DB 結構，但會影響文件與命名一致性。
- 範例：projectflow / 專案管理系統。
- 備註：名稱最好一開始就先定。

## 資料項目 2
- 資料名稱：主要使用者類型
- 這是什麼資料：誰會實際登入並使用系統。
- 為什麼需要這筆資料：會影響角色權限、首頁設計、流程安排。
- 是否必填：是。
- 如果缺少，會造成什麼影響：系統可能做得功能很多，但不符合真正使用者需求。
- 最晚要在哪個階段補齊：需求定義階段。
- 這筆資料會不會影響正式系統結構：會，因為會影響角色與權限設計。
- 範例：管理者、專案執行者、主管、外部廠商。
- 備註：如果使用者類型很多，至少先列出第一版主要角色。

## 資料項目 3
- 資料名稱：實際流程範例
- 這是什麼資料：一筆真實或接近真實的使用案例，描述事情怎麼從開始走到結束。
- 為什麼需要這筆資料：沒有流程範例，系統很容易只停在抽象討論。
- 是否必填：是。
- 如果缺少，會造成什麼影響：頁面與流程設計容易憑想像，後面會反覆修改。
- 最晚要在哪個階段補齊：探索階段開始前。
- 這筆資料會不會影響正式系統結構：會，因為流程會影響狀態與欄位設計。
- 範例：收到新案件 → 建立專案 → 整理需求 → 核准 → 派工 → 執行 → 結案。
- 備註：至少提供 1～2 個完整案例。

## 資料項目 4
- 資料名稱：測試資料 / 假資料範圍
- 這是什麼資料：在前期探索與測試時，可接受使用哪些示範資料。
- 為什麼需要這筆資料：前端探索一定需要資料支撐，不然需求方無法有效驗。
- 是否必填：建議填。
- 如果缺少，會造成什麼影響：畫面只能空著看，流程很難被確認。
- 最晚要在哪個階段補齊：探索階段前。
- 這筆資料會不會影響正式系統結構：通常不直接影響，但會影響驗收效率。
- 範例：假專案名稱、假廠商、假任務內容。
- 備註：要先說清楚哪些是假資料、哪些是真實資料。

## 資料項目 5
- 資料名稱：正式輸出需求
- 這是什麼資料：系統最後要輸出什麼結果，例如報表、文件、列表、狀態摘要。
- 為什麼需要這筆資料：很多系統前面流程做得很細，但最後不知道輸出要長什麼樣。
- 是否必填：是。
- 如果缺少，會造成什麼影響：前面都做完了，最後交付形式卻不符合需要。
- 最晚要在哪個階段補齊：正式化前。
- 這筆資料會不會影響正式系統結構：會，因為會影響欄位與資料聚合方式。
- 範例：專案列表、未付款總表、廠商對帳摘要、結案報表。
- 備註：若有固定格式，最好一開始就附樣本。

# 模組 5：環境與帳號清單（範例填寫）

## 環境 1
- 環境名稱：local
- 這個環境的用途：工程端本機開發與初步測試。
- 前端網址：本機網址（例如 localhost）。
- 使用的平台：本機開發環境。
- GitHub repo：正式 repo。
- Vercel project：通常不直接對應。
- Supabase project：可接 staging DB 或本機測試 DB。
- 需要哪些帳號 / 權限：工程端本機權限、repo 存取權。
- 誰提供這些權限：專案 owner / 管理者。
- 誰負責管理：執行者自己管理本機，但正式權限由 owner 控制。
- 備註：這個環境通常不給需求方直接看。

## 環境 2
- 環境名稱：MIM / 測試站
- 這個環境的用途：日常修改、需求確認、驗收測試。
- 前端網址：MIM 測試站網址。
- 使用的平台：GitHub + Vercel staging / test runtime。
- GitHub repo：正式 repo 的 main / 測試主線。
- Vercel project：測試站專案。
- Supabase project：staging DB。
- 需要哪些帳號 / 權限：需求方可看網址，執行者需 repo / deploy / DB 權限。
- 誰提供這些權限：專案 owner。
- 誰負責管理：owner 與執行者。
- 備註：這一層是需求方主要看的地方，不要再額外多切一層無意義測試站。

## 環境 3
- 環境名稱：production
- 這個環境的用途：正式使用者實際操作。
- 前端網址：正式網域，例如 `pmis.kuya.tw`。
- 使用的平台：正式站部署環境。
- GitHub repo：正式候選來源 repo。
- Vercel project：正式站專案。
- Supabase project：production DB。
- 需要哪些帳號 / 權限：正式部署權限、正式 DB 權限、正式網域管理權限。
- 誰提供這些權限：專案 owner / 公司管理方。
- 誰負責管理：正式維運責任人。
- 備註：只有在 MIM 驗過、修法已確定後，才進正式站。

# 模組 6：驗收標準表（範例填寫）

## 驗收項目 1
- 驗收名稱：登入功能驗收
- 這個驗收在驗什麼：一般使用者能否成功進入系統。
- 成功條件：使用正確帳密可成功登入，登入後可進首頁並維持 session。
- 失敗條件：帳密正確卻無法登入、登入後馬上掉回登入頁、重新整理後 session 消失。
- 驗收方式（手動 / 自動）：手動 + 可補 smoke test。
- 驗收時要附什麼證據：登入成功截圖 / 影片、成功導向頁面證明。
- 誰負責驗收：需求方 + 執行者共同確認。
- 備註：若正式產品模式啟用，也要確認不再被外層多重驗證卡住。

## 驗收項目 2
- 驗收名稱：專案建立流程驗收
- 這個驗收在驗什麼：能否從新建專案一路走到需求整理完成。
- 成功條件：可成功建立專案、看到專案出現在列表、進入專案明細後資料一致。
- 失敗條件：建完後列表看不到、明細資料錯誤、重新整理後消失。
- 驗收方式（手動 / 自動）：手動主驗，可補流程測試。
- 驗收時要附什麼證據：操作錄影、列表與明細對照截圖。
- 誰負責驗收：需求方與執行者。
- 備註：這類驗收要驗 list + detail，不可只看單頁。

## 驗收項目 3
- 驗收名稱：角色權限驗收
- 這個驗收在驗什麼：不同角色能不能只看到自己該看的內容。
- 成功條件：管理者能看全域；一般執行者只能看自己權限範圍；外部角色看不到內部管理功能。
- 失敗條件：低權限看得到高權限功能、角色切換後資料暴露。
- 驗收方式（手動 / 自動）：手動為主，必要時補權限測試。
- 驗收時要附什麼證據：不同帳號登入結果截圖。
- 誰負責驗收：管理者 + 執行者。
- 備註：正式站前一定要驗這一項。

## 驗收項目 4
- 驗收名稱：環境驗收
- 這個驗收在驗什麼：MIM 與正式站是否分清楚、網址是否正常、部署後頁面是否可開。
- 成功條件：MIM 可供日常驗收、正式站網址可正常開啟、環境用途清楚分離。
- 失敗條件：測試站與正式站混用、需求方看不到驗收站、正式站指錯資料庫。
- 驗收方式（手動 / 自動）：手動盤點為主。
- 驗收時要附什麼證據：網址列表、環境對應說明、必要時附部署紀錄。
- 誰負責驗收：owner / 維運 / 執行者。
- 備註：這一項是避免環境混掉的基本保護。

## 一句話總結

> 這份範例的重點，是讓使用者看懂：模組 4～6 不是只是列資料名稱，而是要把「到底要準備哪些東西、在哪個環境下做、以及什麼才算真的驗收過」講到足夠明確，別人才接得住。
