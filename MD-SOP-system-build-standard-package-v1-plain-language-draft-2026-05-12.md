# MD-SOP — 系統建置 SOP 標準包 v1（白話版草案）— 2026-05-12

Status: draft
Purpose: 把「如何從 0 開始建立一套新系統」整理成一般使用者也能看懂、能填寫、能交接給執行者的白話 SOP 標準包。

---

## 0. 這份文件要解決什麼問題

很多人知道自己想做一套系統，
但不知道該準備什麼資料，也不知道要怎麼描述，
最後就會變成：

- 一直口頭補充
- 做到一半才發現需求沒講清楚
- 使用者看不到畫面，無法有效確認
- 系統反覆修改很多次
- 每次都像在重做 MVP

這份 SOP 的目的，就是把這整件事拆成清楚的表單與步驟。

讓未來的新使用者就算不懂工程，
也能照著回答問題，把需要的資料交齊。

---

## 1. 這套 SOP 最後會包含哪些模組

這套標準包分成 10 個模組。

1. 名詞定義表
2. 角色與權限表
3. 流程與狀態表
4. 資料蒐集清單
5. 環境與帳號清單
6. 驗收標準表
7. 變更管理表
8. 階段進出場條件表
9. 環境分層規則表
10. 交接與維運責任表

每一份都要用白話寫，讓一般人看得懂。

# 模組 1：名詞定義表

## 這份表是做什麼的

這份表用來避免大家講的是同一個詞，心裡想的卻不是同一件事。

例如：
- 專案是什麼？
- 任務是什麼？
- 確認是什麼？
- 完成和結案有什麼差別？

如果這些詞沒有先講清楚，後面很容易做錯。

## 使用者要回答的問題（白話）

### 1. 這個詞是什麼意思？
請用你自己的話解釋，不要只寫很短。

### 2. 這個詞會出現在什麼情境？
例如是在建立資料時、審核時、完成時，還是查詢時。

### 3. 這個詞和哪個詞很像，但其實不一樣？
例如「完成」和「結案」是不是不同？如果不同，差在哪？

### 4. 這個詞是不是正式狀態？
例如「暫存」和「正式確認」就可能是不同層級。

## 欄位設計

- 名詞名稱
- 白話解釋
- 會出現在哪個情境
- 容易混淆的詞
- 與相似詞的差別
- 是否屬於正式狀態 / 正式資料
- 備註

# 模組 2：角色與權限表

## 這份表是做什麼的

這份表要回答：

- 誰可以做什麼
- 誰可以看什麼
- 誰不能碰什麼
- 哪些動作只有特定角色能做

## 使用者要回答的問題（白話）

### 1. 這套系統裡有哪些人？
例如：管理者、主管、內部執行者、外部廠商、客戶。

### 2. 每一種人平常要做什麼？
請用工作內容去描述，不要只寫職稱。

### 3. 每一種人可以看到哪些資料？
例如只能看自己的、能看全部的、只能看已確認的。

### 4. 每一種人可以修改哪些資料？
例如可以新增但不能刪除，或可以編輯但不能確認。

### 5. 哪些動作一定要由特定角色操作？
例如：正式確認、核准付款、結案、刪除。

## 欄位設計

- 角色名稱
- 這個角色平常負責什麼
- 可查看內容
- 可新增內容
- 可修改內容
- 可刪除內容
- 可確認 / 核准的動作
- 不可操作的項目
- 備註

---

# 模組 3：流程與狀態表

## 這份表是做什麼的

這份表用來說清楚一件事怎麼開始、怎麼往下走、什麼時候算完成。

也要說清楚：
- 如果做錯了怎麼辦
- 如果中途取消怎麼辦
- 如果做完後又要重開怎麼辦

## 使用者要回答的問題（白話）

### 1. 一件事情從開始到結束，會經過哪些步驟？
請用順序寫出來。

### 2. 每個步驟完成後，下一步是什麼？
如果有多種情況，也請分開寫。

### 3. 哪一刻算「正式確認」？
這非常重要，因為正式確認後通常規則會改變。

### 4. 正式確認後，還能不能改？
如果能改，誰可以改？怎麼改？

### 5. 如果流程中斷、取消、做錯、重做，要怎麼處理？
這些例外情況一定要寫。

## 欄位設計

- 流程名稱
- 目前步驟
- 誰負責這一步
- 進入這一步的條件
- 完成這一步後會去哪一步
- 這一步是否可修改
- 這一步是否屬正式狀態
- 例外情況處理方式
- 備註

# 模組 4：資料蒐集清單

## 這份表是做什麼的

這份表要列出：

- 新專案開始前，到底要先準備哪些資料
- 哪些是一定要有的
- 哪些可以之後再補
- 哪些缺了就不能開工

## 使用者要回答的問題（白話）

### 1. 這個系統開始前，一定要先知道什麼？
例如：名稱、對象、流程、帳號、範例資料。

### 2. 哪些資料沒有的話，會完全沒辦法做？
這些要列成必填。

### 3. 哪些資料可以先空著，之後再補？
避免一開始被太多欄位卡死。

### 4. 哪些資料只影響前期看流程？
例如假資料、示意文字、草圖。

### 5. 哪些資料會影響正式系統結構？
例如角色、欄位、狀態、正式輸出格式。

## 欄位設計

- 資料項目名稱
- 這是什麼資料
- 為什麼需要這筆資料
- 是否必填
- 缺少時會造成什麼影響
- 可在哪個階段補齊
- 是否影響正式系統結構
- 範例

---

# 模組 5：環境與帳號清單

## 這份表是做什麼的

這份表用來確認：

- 使用者要在哪裡看前端
- 系統在哪裡部署
- 資料庫在哪裡
- 需要哪些帳號和權限
- 誰要提供 GitHub / Vercel / Supabase 的資料

## 使用者要回答的問題（白話）

### 1. 使用者要去哪裡看這套系統？
請提供前端網址或未來要使用的平台。

### 2. 這個專案會用哪些服務？
例如 GitHub、Vercel、Supabase。

### 3. 這些服務的帳號和權限由誰提供？
要明確，不然後面會卡住。

### 4. 有沒有分測試環境和正式環境？
如果有，要清楚寫出來。

### 5. 誰可以登入這些環境？
例如誰能看 preview、誰能進資料庫、誰能改 production。

## 欄位設計

- 環境名稱（local / preview / staging / production）
- 用途
- 前端網址
- 使用平台
- GitHub repo
- Vercel project
- Supabase project
- 需要的帳號 / 權限
- 提供者
- 管理者
- 備註

# 模組 6：驗收標準表

## 這份表是做什麼的

這份表要回答：

- 怎樣才算完成
- 怎樣不算完成
- 要驗什麼
- 誰來驗
- 驗收時要拿出什麼證據

## 使用者要回答的問題（白話）

### 1. 你怎麼判斷這功能真的可用？
請不要只寫「看起來正常」。

### 2. 使用者操作時，最重要的成功條件是什麼？
例如：能送出、能查到、能同步、能回頭修改。

### 3. 哪些畫面或資料一定要同步一致？
例如列表、明細、首頁摘要、匯出結果。

### 4. 有哪些錯誤是不能接受的？
例如：資料遺失、權限亂掉、同步錯誤。

### 5. 驗收時要附什麼證據？
例如：截圖、影片、網址、測試結果、變更清單。

## 欄位設計

- 驗收項目
- 驗收目的
- 成功條件
- 失敗條件
- 驗收方式（手動 / 自動）
- 驗收證據
- 驗收人
- 備註

---

# 模組 7：變更管理表

## 這份表是做什麼的

系統做出來後，一定還會改。

這份表用來區分：
- 這是 bug
- 這是小調整
- 這是新功能
- 這會不會影響既有流程

## 使用者要回答的問題（白話）

### 1. 這次想改的是什麼？
請描述具體，不要只說「想優化一下」。

### 2. 這個改動是修錯，還是新增需求？
兩者差很多。

### 3. 這個改動會影響哪些地方？
例如流程、資料、權限、畫面、報表。

### 4. 改了之後要重驗哪些內容？
不能只驗改到的那一頁。

### 5. 這次改動要不要重新跑探索流程？
如果影響大，就不能直接硬改。

## 欄位設計

- 變更名稱
- 變更類型（bug / 優化 / 新功能 / 結構調整）
- 變更原因
- 影響範圍
- 需要重驗的項目
- 是否需重開探索
- 是否影響正式資料結構
- 備註

---

# 模組 8：階段進出場條件表

## 這份表是做什麼的

這份表是為了避免事情一直卡在中間。

例如：
- 什麼情況才可以開始做探索？
- 什麼情況才可以進正式化？
- 什麼情況才可以進 staging 驗收？
- 什麼情況才可以上正式環境？

## 使用者要回答的問題（白話）

### 1. 哪些條件滿足後，才能進下一階段？
請列清楚，不然會一直邊做邊猜。

### 2. 哪些條件沒滿足，就不能往下走？
這些是卡控點。

### 3. 哪些文件或資料一定要先補齊？
例如角色表、流程表、環境權限。

### 4. 哪些驗證一定要過？
例如前端可見、流程確認、資料同步正確。

## 欄位設計

- 階段名稱
- 進入條件
- 不可缺少的資料
- 必過驗證項
- 可交付產物
- 可否進下一階段
- 備註

---

# 模組 9：環境分層規則表

## 這份表是做什麼的

這份表用來規定：

- 哪個環境是做探索的
- 哪個環境是做測試驗收的
- 哪個環境是正式使用的
- 哪些資料不能混用

## 使用者要回答的問題（白話）

### 1. 這個環境是拿來做什麼的？
例如：只給內部看、給需求方驗收、給正式使用者操作。

### 2. 這個環境可以放什麼資料？
例如測試資料、假資料、正式資料。

### 3. 這個環境不能做什麼？
例如不能拿來跑正式營運、不能接 production DB。

### 4. 這個環境和其他環境怎麼分開？
避免資料與權限混掉。

## 欄位設計

- 環境名稱
- 用途
- 允許資料類型
- 禁止事項
- 對應平台
- 是否對外可見
- 是否可給需求方操作
- 備註

# 模組 10：交接與維運責任表

## 這份表是做什麼的

系統做完不是結束，還要有人接、有人管、有人修。

這份表用來回答：
- 誰接手維運
- 誰負責帳號
- 誰負責部署
- 誰負責資料庫
- 出問題找誰

## 使用者要回答的問題（白話）

### 1. 系統上線後誰負責日常管理？

### 2. 誰有正式環境的權限？

### 3. 誰負責備份、部署、資料庫、網域？

### 4. 如果原本的人不在了，誰接手？

### 5. 如果系統出錯，第一時間找誰？

## 欄位設計

- 項目（部署 / 網域 / DB / 帳號 / 驗收 / 維運）
- 主責人
- 協作人
- 備援接手人
- 權限位置
- 資料存放位置
- 緊急處理方式
- 備註
