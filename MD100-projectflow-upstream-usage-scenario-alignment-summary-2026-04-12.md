# MD100 — projectflow upstream usage scenario alignment summary (2026-04-12)

> Status: ACTIVE
> Note: 本文件用來整理目前已核對過的 `projectflow` 上游使用情境、功能責任與資料承接判斷，供後續逐區盤點與回查使用。


## 1. 文件定位

本文件用來把目前已核對過的 `projectflow` 上游使用情境，整理成一份結構化摘要。

本文件目的：
- 固定目前對上游使用情境的共識
- 區分哪些區塊已相對站穩，哪些仍值得後續精驗
- 作為後續繼續盤點下游三條線時的上游基準文件

---

## 2. 建立專案 → 專案正式成立 → 同步進入列表

### 使用情境
- 在首頁 / 上游入口先建立專案
- 填寫專案基本資料後，專案正式成立
- 專案成立後，應同步出現在：
  - `專案列表頁`
  - `報價及預算列表`

### 判斷
- 功能邏輯：正確
- 責任邊界：正確
- 正式語意：專案一旦成立，就成為整個系統的正式 project 主體；`/projects` 與 `/quote-costs` 都應承接同一批專案，只是視角不同

### 目前狀態
- `/projects`：相對站穩
- `/quote-costs`：大致承接成功，但仍建議持續從正式真值角度驗證

---

## 3. 專案 Detail 上半區：活動資訊 / 客戶資訊 / 需求溝通

### 使用情境
1. `活動資訊`
   - 可自由編輯
2. `客戶資訊`
   - 可自由編輯
3. `需求溝通`
   - 可直接輸入文字並儲存
   - 以 list 呈現
   - 每次新增 / 更新都記錄時間
   - 最新內容往上排

### 判斷
- `活動資訊`：功能正確，且相對站穩
- `客戶資訊`：功能正確，且相對站穩
- `需求溝通`：功能正確，但 DB formalization、timestamp 與排序 readback 建議後續精驗

### 正式語意
- 上半區是正式專案主檔 + 溝通紀錄區
- 不是單純展示頁，而是可持續維護的專案主控資料區

---

## 4. 專案 Detail 下半區：任務發布區

### 使用情境
1. `樹狀結構`
   - 可新增主項目
   - 可新增子項目
   - 採 inline 展開
2. `Excel 匯入`
   - 作為任務發布資料提供層
   - 不是正式 financial / quotation 真值來源
3. `任務交辦`
   - 可對主項目 / 子項目交辦：
     - 設計
     - 備品
     - 廠商
4. `交辦形式`
   - 採側抽屜
5. `三條線差異`
   - 設計 / 備品：此階段不用先指定廠商
   - 廠商：此階段就必須指定廠商

### 判斷
- 樹狀任務發布區：正確
- Excel 匯入作為任務資料提供：正確
- 三條任務交辦：正確
- 設計 / 備品不先指定 vendor：正確
- 廠商線此階段就要指定 vendor：正確，且應視為已拍板規則

### 目前狀態
- 整體相對站穩
- 不是目前 major partial DB-first 主風險區

### 正式語意
- 任務發布區是三條線的共同上游
- Excel 匯入只是任務資料提供，不可漂移成正式 financial / quotation 真值來源

---

## 5. 上游 vendor 交辦補充規則

### 使用情境補充
在 `交辦廠商` 時：

> **必須先選擇廠商，而且該廠商必須已存在於廠商資料清單中，才能被選擇。**

也就是：
- 不接受自由文字 vendor 作為正式主體
- 任務發布區只能使用正式 vendor master

### 判斷
- 功能邏輯：正確
- 責任邊界：正確
  - `廠商資料清單` = vendor identity 主檔
  - `任務發布區` = 使用既有 vendor master，不重新定義 vendor

### 正式語意
這條規則會直接強化：
- vendor assignment
- vendor grouping
- package
- vendor financial

都必須承接同一個 vendor identity。

### 目前狀態
- 規則正確
- 並且會直接提高 Vendor Data / vendor assignment / vendor detail same-source closure 的重要性

---

## 6. 專案 Detail 最後一區：任務檢視板（Tab）

### 使用情境
任務檢視板包含三個 Tab：
- 設計
- 備品
- 廠商

其中：
1. `設計 / 備品`
   - 以任務為主體
   - 檢視有哪些任務
2. `廠商`
   - 以廠商為主體
   - 檢視該廠商底下有哪些任務

### 判斷
- 設計 / 備品以任務為主體：正確
- 廠商以廠商為主體：正確
- 這個差異符合三條線責任差異，應視為固定規則，不應漂移

### 目前狀態
- 設計 / 備品：相對站穩
- 廠商：邏輯正確，且已大致承接主線；後續仍可從 same-source 角度再驗

## 7. 上游總結

### 已清楚且相對健康的部分
1. 建立專案 → 專案正式成立
2. 專案同步進 `/projects` 與 financial 視角列表
3. Detail 上半區活動資訊 / 客戶資訊
4. 任務發布區的樹狀結構與三條任務交辦邏輯
5. 任務檢視板三條線的視角差異
6. vendor 必須先存在於廠商資料清單，才能在交辦廠商時被選擇

### 後續較值得補做精驗的部分
1. `需求溝通`
   - DB readback
   - timestamp
   - 排序一致性
2. `報價及預算列表` 對 project 主體的真值承接
3. vendor identity 的 same-source closure
