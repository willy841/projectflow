# 1. 本次討論結論總覽

本次正式收斂的核心方向是：

> **`project detail` 不再承擔三條線的完整深操作工作台責任，而是回到「專案總覽 + 任務發布 + 導流中心」定位。**

對應重切結果：

- `project detail`
  - 保留：專案背景總覽、execution tree / 任務發布、三條線任務摘要與導流
  - 移除主控責任：深度回覆、確認、整理、文件生成、package 深操作
- `design-tasks`
  - 承接設計線深操作主控責任
- `procurement-tasks`
  - 承接備品線深操作主控責任
- `廠商發包版`
  - 承接 vendor 線處理層主控責任
- `vendor-packages` / package 主線
  - 承接 vendor 線的整理層 / 文件層主責任

# 2. 高階定位重切

## 2.1 `project detail` 的新正式定位

`project detail` 正式定位為：

> **單專案主控台 / 專案總覽頁 / 任務發布入口 / 各工作流導流中心**

不再定位為：
- 三條線共用的完整工作台
- 回覆處理主場
- 文件整理主場
- 文件生成主場
- package post-issue 主場

## 2.2 左選單三條主線頁的正式定位

### A. `/design-tasks`
定位：
> **設計線主控工作台**

承接：
- 設計任務深度查看
- 設計回覆管理
- 回覆確認
- 設計文件整理
- 設計文件生成 / 重新生成

### B. `/procurement-tasks`
定位：
> **備品線主控工作台**

承接：
- 備品任務深度查看
- 備品回覆管理
- 回覆確認
- 備品整理
- 備品文件生成 / 重新生成

### C. `廠商發包版`
定位：
> **vendor 線主控工作台 / 發包處理主場**

承接：
- vendor 任務總表
- 單筆 vendor 任務處理
- 發包確認
- 從任務層導入 package 主線

### D. `/vendor-packages` 與 package 主線
定位：
> **vendor 線的整理層 / 文件層主場**

承接：
- 已發包內容的 package 整理主線
- package 內容整理
- 文件生成 / 重新生成
- package detail 深操作

# 3. `project detail` 保留責任

## 3.1 專案背景總覽層
保留：
- 專案標題 / 專案狀態
- 活動資訊
- 客戶資訊
- 成本 / 預算摘要
- 需求溝通

這些屬於單專案主控台必要背景資訊，不屬於任何單一工作流主控頁。

## 3.2 任務發布層
保留：
- execution tree
- 主項目 / 子項目樹
- 新增主項目
- Excel 匯入
- 從 execution item 發起三條線任務：
  - 設計
  - 備品
  - 廠商

正式語意：
> 任務發布責任仍屬於 `project detail` 核心，不外移。

## 3.3 三條線摘要與導流層
保留：
- 專案設計摘要與導流
- 專案備品摘要與導流
- 專案廠商任務摘要與導流

正式語意：
> `project detail` 要看得到專案內目前各條線發生了什麼，但不再在同頁承擔完整深操作。

# 4. 設計 / 備品在 `project detail` 的正式邊界

## 4.1 設計 / 備品區的正式定位

`project detail` 內的設計 / 備品區，正式定位為：

> **單專案內的任務摘要清單 + 每筆任務導流入口**

不是：
- 工作台
- 回覆主場
- 文件整理主場
- 文件生成主場

## 4.2 每筆任務保留欄位（已拍板）

每筆設計任務 / 備品任務，在 `project detail` 只保留：
- 任務標題
- 狀態
- 導流入口（每筆任務）

補充定義：
- 下方檢視區不需要再帶母卡欄位
- 不需要顯示主項目 / 子項目來源
- 任務檢視區的角色是「已發布任務清單 + 導流」，不是資訊展開區

## 4.3 明確不保留的內容（已拍板）

`project detail` 的設計 / 備品區不再保留：
- 回覆列表
- 回覆確認
- 整理區
- 文件生成操作
- 深度展開工作區

## 4.4 深操作責任承接頁

### 設計
正式由：
- `/design-tasks`
承接深操作主責任。

### 備品
正式由：
- `/procurement-tasks`
承接深操作主責任。

# 5. Vendor 在 `project detail` 的正式邊界

## 5.1 Vendor 區的正式定位

`project detail` 內的 vendor 區，正式定位為：

> **單專案內的 vendor 任務摘要清單 + 每筆任務導流入口**

此定位需與設計 / 備品兩條線保持一致。

因此它不是：
- vendor 工作區
- package 主線
- 文件整理主場
- 文件生成主場

## 5.2 每筆 vendor 任務保留欄位（已拍板）

每筆 vendor 任務，在 `project detail` 比照設計 / 備品，只保留：
- 任務標題
- 狀態
- 導流入口

正式導流規則：
- 導向左選單 `廠商發包版`
- 並應比照設計 / 備品，採：頁面 + project filter + task 定位

補充定義：
- 不需要顯示主項目 / 子項目來源
- 不需要在這裡帶出大量補充欄位
- vendor 區同樣是「已發布任務清單 + 導流」，不是工作區

## 5.3 明確不保留的內容（已拍板）

`project detail` 的 vendor 區不再保留：
- 實際處理工作區
- 大量補充欄位編輯
- package 內容整理
- 文件生成 / 重新生成
- post-issue 主工作區

## 5.4 正式語意

一句話定義：

> `project detail` 的 vendor 區與設計 / 備品一致，只保留任務摘要與導流；vendor 真正的處理主線在發包確認後，轉入 package 整理 / 文件主線。

# 6. `vendor-packages` 的正式邊界（vendor 線新版）

## 6.1 `vendor-packages` 的正式定位

`vendor-packages` 不再理解為 vendor 任務主工作台，而是：

> **vendor 線的整理層 / 文件層主場**

也就是承接：
- 發包確認後的內容整理
- package 內容整理
- 文件生成 / 重新生成
- 文件查看

## 6.2 `vendor-packages` list 頁的正式語意

`/vendor-packages` 列表頁顯示的，不是「仍在處理中的 vendor 任務」，而是：

> **仍有整理、生成、查看需求的 package 容器**

因此這頁的狀態主軸應理解為：
- 文件未生成
- 文件已生成
- 文件需更新

而不是把 package 當另一套 issue 工作區狀態機。

## 6.3 已結案 / 歷史邊界

本輪目前只先定：
- `vendor-packages` 不是 vendor issue 工作區
- package 的歷史 / 留存邊界，需與 `closeout` / 專案結案邏輯一起討論

換句話說：
- issue 層完成點
- project 層 closeout
- package 層文件留存

這三者不能混成單一「已結案」語意。

## 6.4 package detail 的相容查閱能力

正式規則：
- package detail 可保留相容查閱能力
- 若從歷史 / 結案脈絡打開某個 package detail，仍可查看其留存內容

## 6.5 一句話總結

> `vendor-packages` 是 vendor 線的整理層 / 文件層，不是 vendor 任務工作區；後續需再與 closeout 邏輯一起收斂其歷史邊界。

## 5.5 2026-04-05 新增定稿：Project Detail 的任務發布 / 檢視使用情境

### A. 使用情境
- 使用者在 `Project Detail` 的任務發布區中，先展開主項目或子項目
- 可從主項目獨立發布任務
- 也可從子項目獨立發布任務
- 任務一旦發布出去，下方任務檢視區就只負責顯示「目前已發布了哪些任務」
- 真正承接後續處理、編輯、確認、整理、文件生成的工作，交給各自未來的主控頁

### B. 上半部正式責任
`Project Detail` 上半部是 **任務發布區**，正式負責：
- 顯示主項目 / 子項目結構
- 展開主項目 / 子項目
- 從主項目獨立發布任務
- 從子項目獨立發布任務
- 將任務發布到三條線：
  - 設計
  - 備品
  - 廠商

補充定義：
- 任務發布單位是「主項目」或「子項目」本身
- 發布責任留在 `Project Detail`
- `Project Detail` 仍是單專案任務發布主場

### C. 下半部正式責任
`Project Detail` 下半部是 **已發布任務檢視區**，正式負責：
- 顯示已發布的設計任務
- 顯示已發布的備品任務
- 顯示已發布的廠商任務
- 提供每筆任務的導流入口，導到未來真正承接的工作台 / 主控頁

正式語意：
> 下半部是「已發布任務清單 + 導流區」，不是工作區、不是編輯區。

### D. 下半部每筆任務保留欄位
`Project Detail` 下半部每筆任務只保留：
1. 任務標題
2. 狀態
3. 導流入口

### E. 下半部明確不顯示的內容
`Project Detail` 下半部不需要顯示：
- 主項目 / 子項目來源
- 尺寸
- 材質
- 數量
- 需求說明
- 母卡完整欄位
- 回覆列表
- 回覆確認
- 文件整理資訊
- 文件生成資訊
- 其他深操作欄位

### F. 下半部明確不負責的事情
`Project Detail` 下半部不負責：
- 編輯任務
- 處理回覆
- 確認回覆
- 整理文件
- 生成文件
- 發包後 package 深操作

## 6.6 設計任務板正式承接 spec（2026-04-05 新增定稿）

### A. 正式定位
`設計任務板` 是：

> **設計執行者在做整體檢視、任務處理與文件收束時的主要工作區。**

它不是跨專案任務大雜燴總表，也不是把所有任務直接攤平成單層列表；正式結構應分成：
1. 專案入口層
2. 單專案任務列表層
3. 單任務詳情處理頁
4. 設計文件整理頁（即最終文件頁）

### B. 第一層：專案入口層
第一層應先以 **專案** 作為分流入口，避免不同專案任務混在一起造成混亂。

每個專案只保留：
1. 專案名稱
2. 任務數量
3. 活動日期
4. 主 CTA

正式語意：
> 第一層只是專案分流入口，不是專案摘要分析頁。

明確不需要顯示：
- 客戶名稱
- 回覆狀態摘要
- 文件狀態摘要
- 專案代碼
- 成本資訊
- 額外說明文
- 其他次要輔助欄位

### C. 第二層：單專案任務列表層
點進專案後，才看到該專案底下的各自設計任務。

每筆任務首屏只保留：
1. 任務標題
2. 尺寸
3. 材質
4. 結構
5. 數量

以下資訊不放在列表首屏，改由更深一層處理：
- 需求說明
- 參考連結

正式語意：
> 任務列表首屏是快速判讀任務基本規格的工作列表，不在這一層攤開完整需求背景。

### D. 第三層：設計任務詳情頁
設計任務詳情頁是真正的任務處理頁。

#### D-1. 原始任務資訊區
首屏顯示以下 7 個欄位：
1. 任務標題
2. 尺寸
3. 材質
4. 結構
5. 數量
6. 需求說明
7. 參考連結

正式語意：
> 任務列表層負責快速判讀；任務詳情頁則負責完整顯示原始需求。

#### D-2. 「執行處理」
在設計任務詳情頁裡，這一塊不再叫「回覆區」，而是：

> **針對這項需求的「執行處理」**

正式語意：
- 這裡的重點不是 chat / reply 紀錄
- 而是這項需求已進入實際執行處理
- 它是任務處理方案管理區，不是留言串

#### D-3. 每筆處理方案欄位
每一個獨立的處理方案都要有：
1. 標題
2. 尺寸
3. 材質
4. 結構
5. 數量
6. 金額
7. 預覽位置
8. 執行廠商

正式語意：
> 每一筆處理方案都是一筆完整的設計處理方案資料，而不是零碎回覆。

#### D-4. 處理方案之間的關係
正式定義：
> **每一筆處理方案都是獨立的處理方案，彼此之間沒有層級之分。**

也就是：
- 平行列表
- 沒有父子層級
- 沒有樹狀回覆
- 不是 thread / comment 結構

#### D-5. 單筆儲存與整區確認
每一個獨立的處理方案都要有一個：
- `儲存` 按鈕

按下單筆 `儲存` 的正式語意：
> **該處理方案已經儲存成功。**

補充規則：
- `儲存` 只代表單筆方案資料保存成功
- `儲存` 不等於正式成立進文件

整個處理方案區則需要有一個：
- `確認` 按鈕

按下整區 `確認` 的正式語意：
> **目前這個處理方案區裡的所有處理方案，全部正式成立並一起收進文件內。**

補充規則：
- 確認單位是整區，不是逐筆確認
- 同一筆設計需求底下，可以有多筆獨立處理方案同時成立
- 文件來源不是單筆挑選，而是該需求目前確認成立的整組處理方案

#### D-6. 確認後再修改的規則
正式規則：
> **處理方案區一旦在確認後又被新增、修改或刪除，必須重新確認，才能再進文件。**

也就是：
- 確認只對當下那一版區內內容有效
- 一旦區內內容改變，原本進文件的資格先失效
- 必須重新確認後，才重新成為文件來源

#### D-7. 2026-04-05 新增定稿：執行處理區按鈕層級規則
正式規則：
- `新增執行處理` 與 `全部確認` 屬於**整區層級操作**
- 應放在 `執行處理` header 同一列的 action 區
- 每一筆處理方案卡片內只保留：
  - `儲存`
  - `刪除這筆處理`
- 不保留單筆 `確認`

正式語意：
> **新增與全部確認是整區操作；單筆卡片只負責編輯、儲存與刪除。**

#### D-8. 2026-04-05 新增定稿：確認承接文件規則
正式規則：
- 文件頁承接的是**當次全部確認時，整個執行處理區內的完整內容**
- 不是逐筆確認後各自送進文件
- 單筆 `儲存` 不會把內容送進文件頁
- 只有整區 `全部確認` 才會把目前整區內容承接進文件頁

### E. 第四層：設計文件整理頁（= 最終文件頁 = 終點）
設計任務詳情頁確認之後，不是直接進另一個獨立文件頁，而是先進：

> **設計文件整理頁**

而這個頁面本身就是：
> **最終文件頁，也是整條設計線的終點。**

#### E-1. 頁面定位
這一頁不是再做任務處理，而是：
- 把已確認成立的處理方案整理成文件內容
- 作為最終設計文件本體

#### E-2. 頁面內容形式
頁面內要條列式地顯示每一項處理方案。

每一項需要的欄位如下：
1. 編號
2. 項目
3. 尺寸
4. 材質與結構
5. 數量

#### E-3. 編輯規則
上述欄位都需要具備：
- 可編輯調整功能

但正式規則是：
> **這一層的調整不需要回寫到上一層的任務處理方案裡。**

正式語意：
- 上一層是任務處理方案來源層
- 這一層是文件整理副本層
- 可為了文件呈現做編修，但不回寫上層方案資料

#### E-4. 頁尾欄位
在這個頁面的最後，需要有一個欄位用來放置：
- 檔案位置的連結

#### E-5. 終點規則
正式定義：
> **設計文件整理頁就是最終文件頁，它本身就是終點。**

也就是：
- 不再另外做一層正式設計文件頁
- 不再在整理頁後面再接一個文件輸出頁
- 文件整理與最終文件合併成同一頁

## 6.7 採購備品板正式承接 spec（2026-04-05 新增定稿）

### A. 正式定位
`採購備品板` 是：

> **備品執行者 / 採購執行者在做整體檢視、任務處理與文件收束時的主要工作區。**

高階結構與設計線一致，正式分成：
1. 專案入口層

> 對 `project detail` 做第一階段收口時，最正規的方式不是直接在舊深操作區上大砍，而是先做責任分層，再做渲染替換。

正式採用策略：
- 先新增獨立的 summary renderer 元件
- 先在 `ExecutionTreeSection` 內做 summary selector
- 先讓分類區切到新 renderer
- 舊深操作邏輯暫時保留在檔案中，但不再作為目前分類區主渲染路徑
- 待第一階段穩定後，再進第二輪清理死碼

此策略已驗證為比直接拆舊大段 UI 更安全、更能銜接後續架構調整。
