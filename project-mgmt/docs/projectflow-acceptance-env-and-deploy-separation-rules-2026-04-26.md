# Projectflow acceptance env and deploy separation rules — 2026-04-26

Status: active

## Purpose

This document formalizes the current responsibility boundary between:
- acceptance runtime
- local pre-production / production-local runtime
- deploy packaging path

It exists to prevent acceptance, local env switching, and deploy preparation from being treated as the same operational path.

---

## 1. Runtime classes

### A. Acceptance runtime
Role:
- GitHub `main` aligned validation runtime
- used by Playwright and `tests/formal-acceptance-v2`

Entry path:
- `./scripts/start-acceptance.sh`
- Playwright `webServer.command`

Env source:
- `.env.acceptance`
- copied into `.env.local` before `next dev`

Rules:
- must run with `next dev`
- must not inherit a forced non-standard `NODE_ENV`
- is allowed to use `.env.local` as a compatibility bridge for local Next.js loading
- must be treated as validation-only runtime, not deploy runtime

### B. Production-local runtime
Role:
- local pre-production / official-stack-facing runtime check
- used to verify local official DB path assumptions before formal deploy

Entry path:
- `./scripts/start-production-local.sh`

Env source:
- `.env.production-local`
- copied into `.env.local` before `next dev`

Rules:
- must run with `next dev`
- must not inherit a forced non-standard `NODE_ENV`
- is not the same thing as deploy packaging
- is only a local runtime shape for pre-production verification

### C. Deploy path
Role:
- package and send deployable code to `webapp-deployer`

Entry path:
- `./scripts/deploy.sh prepare`
- `./scripts/deploy.sh deploy`
- `./scripts/deploy.sh status`

Rules:
- deploy path must not depend on `.env.acceptance`
- deploy package must exclude `.env*`
- production container runtime owns `NODE_ENV=production`
- deploy path is not an acceptance runner

---

## 2. Responsibility boundary for `.env.local`

Current reality:
- local Next.js runtime reads `.env.local`
- acceptance and production-local scripts currently use `.env.local` as a compatibility handoff target

Formal boundary:
- `.env.acceptance` is the source of truth for acceptance runtime
- `.env.production-local` is the source of truth for production-local runtime
- `.env.local` is currently only a local runtime bridge file generated by the start script
- `.env.local` must not be treated as the long-term owner of acceptance or production-local semantics

Operational consequence:
- if `.env.local` content differs from the intended source file, the source file wins after the next environment start script run
- future cleanup may replace this bridge model, but current phase only formalizes the boundary

---

## 3. NODE_ENV rule

Accepted rule:
- `next dev` must not run under forced `NODE_ENV=production`
- local shell/runtime pollution may set `NODE_ENV=production`, which causes Next.js non-standard warning in dev runtime
- acceptance and production-local start scripts must explicitly clear inherited `NODE_ENV` before `npm run dev`

Meaning:
- warning removal is a runtime hygiene correction
- it is not a product behavior change
- production deploy runtime still owns `NODE_ENV=production` inside Docker / deploy container path

---

## 4. Separation contract

### Acceptance side may do
- run formal acceptance
- seed / validate GitHub-main-aligned runtime behavior
- use Playwright webServer startup path

### Acceptance side must not be treated as
- deploy packaging path
- formal production runtime
- official stack deployment path

### Deploy side may do
- package application code
- exclude tests and env secrets
- build with production settings inside deploy container path

### Deploy side must not do
- reuse acceptance env files
- depend on local `.env.local` bridge behavior
- be described as the same thing as local acceptance runtime

---

## 5. Current phase conclusion

At the end of this phase:
- acceptance and production-local start paths remain bridge-based via `.env.local`
- this is acceptable for now, as long as the responsibility boundary is explicit
- `NODE_ENV` pollution in local dev startup should be cleared immediately
- deeper replacement of `.env.local` bridge behavior can be deferred to a later runtime/env cleanup phase
