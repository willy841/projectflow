# MD-RULES — projectflow 文件角色與命名語義規則 — 2026-05-12

Status: active
Scope: `projectflow` only
Purpose: 明確定義 `projectflow` 文件命名時各種前綴角色的語義、職責與不可混用邊界

## 1. 為什麼需要這份文件

`projectflow` 的文件數量已經很多。
如果只看名字，像：
- 母檔
- 索引檔
- summary
- governance
- package
- handoff

它們的意思很容易看起來相近，實際卻扮演不同角色。

因此這份文件的目的，是把：
> **各種命名到底代表什麼職責**
正式寫清楚。

未來命名不能只憑感覺，也不能把不同角色混用。

---

## 2. 分辨原則

分辨文件角色時，**不要只看字面意思**。
應以這份文件真正承擔的：
- 角色
- 用途
- 載入時機
- 是否具有決策權
來判定。

也就是說：
> **文件命名應依職責，不應依文風或感覺。**

---

## 3. 各類命名的正式語義

## A. `MASTER`
### 角色
最高層 source-of-truth / 主法源入口

### 用途
- 定義什麼才算當前有效主線
- 定義新舊衝突時，以誰優先
- 定義後續應該依哪些治理入口判斷

### 應該包含
- 主法源判定
- 主線閱讀規則
- 衝突時的優先順序

### 不應該包含
- 大量細節操作步驟
- 細部 deploy 指令大全
- 純摘要內容

### 一句話
> `MASTER` 是最高準據，不是操作手冊，也不是單純摘要。

---

## B. `INDEX`
### 角色
導航地圖 / 路由表 / 文件分層地圖

### 用途
- 告訴你該先讀哪份、再讀哪份
- 分類 active / historical / secondary / missing reference
- 幫助讀者不要迷路

### 應該包含
- 文件清單
- 分層說明
- 入口順序
- 讀法建議

### 不應該包含
- 太多制度規則本體
- 太多執行細節
- 自己變成另一份 `MASTER`

### 一句話
> `INDEX` 負責導航，不負責當最高裁判。

---

## C. `SUMMARY`
### 角色
濃縮結論 / 現況總結 / 高層狀態摘要

### 用途
- 快速理解目前系統狀態
- 快速接手
- 管理者視角總結

### 應該包含
- 關鍵現況
- 結論
- 高層判讀
- 近期主要狀態

### 不應該包含
- 全部細節
- 取代完整治理規則
- 取代完整操作手冊

### 一句話
> `SUMMARY` 是濃縮結論，不是全量規則，也不是地圖。

---

## D. `GOVERNANCE`
### 角色
制度規則 / 邊界治理 / 行為約束文件

### 用途
- 定義什麼可做、什麼不可做
- 定義 staging / production / context boundary
- 定義治理流程與邊界

### 應該包含
- 規則
- 邊界
- 分工
- 流程治理

### 不應該包含
- 純導航用途
- 純事件 handoff
- 純最終摘要

### 一句話
> `GOVERNANCE` 是制度文件，負責把規則寫死。

---

## E. `PACKAGE`
### 角色
整包系統封裝的總入口 / bundle entrypoint

### 用途
- 當整個 `projectflow` 被視為一個封裝系統時，提供單一入口
- 告訴讀者：整包內容有哪些層、何時應打開、何時不應打開

### 應該包含
- 整包最小入口
- 分層總覽
- 封裝用途
- 載入條件

### 不應該包含
- 太細的單一流程操作
- 取代每一份 index / governance / operations 文件

### 一句話
> `PACKAGE` 是整包入口，不是單一規則，也不是單一摘要。

---

## F. `OPERATIONS`
### 角色
操作總手冊 / 執行手冊 / 系統使用方式總說明

### 用途
- 告訴人或 agent 如何實際操作這個系統
- 整理 skills / tools / 路徑 / deploy / DB / 驗收流程

### 應該包含
- 會用到哪些 skill
- 會用到哪些 tool
- deploy 路徑
- GitHub 流程
- staging → production 流程
- 驗收方式

### 不應該包含
- 僅作為最高準據
- 只做純摘要
- 只做 handoff

### 一句話
> `OPERATIONS` 是怎麼做，不是誰說了算。

---

## G. `HANDOFF`
### 角色
交接文件 / 接力棒 / 當下脈絡續接點

### 用途
- 讓下一個 session 接手當下問題
- 說明目前做到哪裡、還差什麼、下一步是什麼

### 應該包含
- 任務目標
- 已完成事項
- 未完成事項
- 真實 blockers
- 下一步建議

### 不應該包含
- 取代長期治理文件
- 取代操作總手冊
- 取代整包封裝入口

### 一句話
> `HANDOFF` 是接力棒，不是憲法，也不是總地圖。

---

## H. `RULES`
### 角色
針對單一主題的命名 / 行為 / 判讀細則

### 用途
- 當某個面向需要精確定義語義或判準時使用
- 比 `GOVERNANCE` 更聚焦、更局部

### 應該包含
- 精確定義
- 用語邊界
- 不可混用規則
- 判定標準

### 不應該包含
- 包山包海的總治理
- 整包入口功能
- 純操作步驟大全

### 一句話
> `RULES` 是局部精準規則，不是整體治理總章。

---

## 4. 不可混用原則

### 1. `MASTER` 不能被拿來當 `OPERATIONS`
如果主要功能是操作手冊，就不應命名成 `MASTER`。

### 2. `INDEX` 不能被拿來當 `GOVERNANCE`
如果主要功能是規定流程與邊界，就不應只叫 `INDEX`。

### 3. `SUMMARY` 不能假裝自己是 source-of-truth
摘要不能取代主法源。

### 4. `HANDOFF` 不能長期主導系統規則
handoff 是時點交接，不是最高長期規則。

### 5. `PACKAGE` 不應淪為普通 index
如果只是列文件清單，那就是 `INDEX`；
只有當它在扮演整包入口時，才應叫 `PACKAGE`。

### 6. `RULES` 不應跟 `GOVERNANCE` 完全重疊
`RULES` 偏局部精準；`GOVERNANCE` 偏整體治理。

---

## 5. 未來命名建議順序

新文件命名時，先問：

1. 這份文件是在當最高準據嗎？
   - 是 → `MASTER`
2. 這份文件是在導航嗎？
   - 是 → `INDEX`
3. 這份文件是在濃縮現況嗎？
   - 是 → `SUMMARY`
4. 這份文件是在寫制度規則嗎？
   - 是 → `GOVERNANCE`
5. 這份文件是在當整包入口嗎？
   - 是 → `PACKAGE`
6. 這份文件是在教怎麼操作嗎？
   - 是 → `OPERATIONS`
7. 這份文件是在做交接嗎？
   - 是 → `HANDOFF`
8. 這份文件是在定義局部精準細則嗎？
   - 是 → `RULES`

若一份文件同時想做很多角色，通常代表它應拆分，而不是硬塞成單一檔。

---

## 6. 一句話總結

> **`projectflow` 文件命名必須依職責，不可依感覺：`MASTER` 管最高準據，`INDEX` 管導航，`SUMMARY` 管濃縮結論，`GOVERNANCE` 管制度邊界，`PACKAGE` 管整包入口，`OPERATIONS` 管怎麼做，`HANDOFF` 管交接，`RULES` 管局部精準規則。若角色混用，應拆檔，不應硬命名。**
