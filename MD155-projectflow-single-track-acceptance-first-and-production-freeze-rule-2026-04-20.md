# MD155 — projectflow 單軌驗收優先與正式站凍住規則 — 2026-04-20

> 目的：正式拍板 `projectflow` 現階段的執行策略，避免繼續在測試站與正式站之間雙軌同步開發，造成判斷與管理成本過高。

---

## 1. 新規則（正式拍板）

使用者已於 2026-04-20 明確同意：

> **現階段 `projectflow` 改採「測試站單軌開發 / 驗收優先」，正式站先凍住；等測試站完成驗收後，再整套搬到正式站。**

這代表：
- 正式站目前不再承接這一輪日常修 bug / 驗收 / 逐項同步
- 正式站此刻的角色，改為**最後正式落地的目標環境**
- 現階段主線全部集中在測試站完成

---

## 2. 角色重新定義

### 2.1 測試站（Acceptance / GitHub）
定位：
- 當前唯一主線
- 所有功能修改、bugfix、驗收、規則收斂都先在這邊完成

用途：
- 日常開發
- 日常驗收
- 功能與流程收斂
- 權限與畫面行為確認

### 2.2 正式站（Production / Local Official System）
定位：
- 現階段先凍住
- 目前不是同步開發環境
- 之後等測試站完成後，再一次正式搬入

用途：
- 本輪暫不承接日常開發與驗收
- 後續最終部署與正式落地用

---

## 3. 正式站現階段規則

### 3.1 正式站可清空
使用者已明確接受：
- 正式站目前資料可清空
- 正式站目前可視為過渡環境
- 不需要為了保留當前過渡資料而持續同步開發

### 3.2 正式站先凍住
在使用者明確宣布「開始搬正式站」之前：
- 不再預設把測試站每個修正都同步到正式站
- 不再預設正式站也要跟著逐項驗收
- 不再讓正式站參與這一輪日常 debug

### 3.3 正式站最後再接環境
待測試站驗收到完整版本後，再一次處理：
- 正式站部署
- 正式 env
- 正式 DB
- 正式 auth
- 正式 runtime / deploy
- 正式站最終驗收

---

## 4. 對 agent 的執行要求

從本文件起，agent 在處理 `projectflow` 時必須遵守：

### 4.1 預設只處理測試站
除非使用者明確要求，否則：
- 所有開發
- 所有 bugfix
- 所有驗收
- 所有規則調整

都先以測試站為唯一主線。

### 4.2 不再預設正式站同步
之後回報時，不可再沿用先前「測試站 + 正式站原則上都要同步」的預設執行方式。

現階段正確預設應改為：
- **先完成測試站**
- **正式站暫不跟進**
- **等使用者批准整套搬移時，再進正式站**

### 4.3 正式站操作需視為下一階段
若後續要動正式站，應視為另一個明確階段，至少要包含：
1. 正式站清空 / reset
2. 正式站部署
3. 正式 env / DB 接線
4. 正式 auth / runtime 驗證
5. 正式站驗收

---

## 5. 為什麼這樣做

這條規則的目的，是降低目前雙站並行帶來的複雜度：
- 不再每修一個 bug 都要判斷是否同步正式站
- 不再讓正式站承接半成品狀態
- 不再讓測試站 / 正式站 / auth / DB / env 的判斷互相污染
- 先把產品本身在測試站做對、做穩
- 最後再處理正式站落地

一句話：

> **先把測試站做完整，再搬正式站，整體管理成本最低。**
