## 2. 資料治理與刪除規則

### 2.1 測試資料治理
正式規則：
- 測試資料一律直接刪除
- 不保留測試樣本
- 不建立測試資料專區
- 未來驗收方式為：
  1. 從前端正式建立一筆專案
  2. 正式走流程測試
  3. 測完直接刪除

### 2.2 刪除語意
正式規則：
- `刪除 = 正式刪除 DB 資料`
- 刪除不是隱藏
- 刪除不是 archive
- 刪除不是只從前端消失

## 3. 專案狀態 / closeout 制度規則

### 3.1 專案主狀態規則
正式規則：
- 新建專案 = `執行中`
- 全部對帳完成 + 全部收款完成 = `已結案`
- homepage / `/projects` / active views 只顯示 `執行中`
- closeout 只顯示 `已結案`

### 3.2 取消結案（reopen）
正式規則：
- 結案資料內必須提供 `取消結案` 按鈕
- 按下後，專案狀態改回 `執行中`
- 若之後要再結案，必須重新按一次結案

### 3.3 取消結案後資料處理
正式規則：
- 取消結案後，**只改專案狀態**
- 不回推 / 不清除：
  - 結案快照
  - 結案紀錄
  - 摘要
- 之後若重新結案，closeout 內容應以當下最新資料**覆蓋**送出

### 3.4 取消結案後 active 視角
正式規則：
- 取消結案後，專案要立刻回到所有執行中視圖，包括：
  - homepage active overview
  - `/projects`
  - recent projects
  - 其他 active lists / active views

### 3.5 Closeout 是否可編輯
正式規則：
- closeout 不可直接編輯
- closeout 是留存 / 顯示區，不是編輯工作區
- 若要改資料，流程為：
  1. 取消結案
  2. 回到執行中主線修改
  3. 再重新結案

## 4. 三條線工作臺 formalization 規則

### 4.1 三條線定位
適用範圍：
- design
- procurement
- vendor

正式規則：
- 三條線要做成**同一家族的正式工作臺**
- 但 formalization 僅限：
  - 前台骨架一致化
  - 視覺節奏一致化
  - 互動節奏一致化
  - 命名語氣一致化
- **不重構 DB schema，不重寫後台核心邏輯**

### 4.2 三條線骨架
正式規則：
- 三條線採**部分固定骨架**
- 固定項：
  1. Header 區
  2. Action 區
  3. 儲存 / 確認 / 返回 / 文件入口的操作節奏
  4. 主要區塊排序邏輯
- 可保留差異項：
  - 中間欄位內容
  - 各線專業欄位與專業資訊結構

### 4.3 儲存 / 確認語意
正式規則：
- `儲存` = 保存目前編輯內容
- `確認` = 正式成立 / 往下游承接的成立點
- 此語意三條線一律固定，不可漂移

### 4.4 文件語意
正式規則：
- 在三條線中，只要叫 `文件`，一律代表：
  - 正式結果頁
  - 正式輸出 / 正式承接後的查看頁
- `文件` 不是暫存頁
- `文件` 不是編輯頁
- `文件` 不是過渡頁

### 4.5 返回動線
正式規則：
- 三條線中的 detail / 文件 / 工作臺頁，都必須保留明確返回入口
- 不可只依賴瀏覽器返回

## 5. UI / wording / visibility guardrails

### 5.1 空狀態語氣
正式規則：
- 空狀態語氣大致統一即可
- 維持：
  - 簡潔
  - 安靜
  - 少系統口吻
- 不要求所有空狀態逐字完全一致

### 5.2 系統生成 code
正式規則：
- system-generated code 不顯示在 UI
- 包含：
  - project code
  - package code
  - child code
  - 其他系統生成辨識碼
- 例外：若未來有明確且必要的產品需求，才可重新討論

### 5.3 人工序號概念
正式規則：
- `#1 / #2 / #3` 這類人工序號概念可保留
- 它們不屬於系統 code 暴露

### 5.4 badge / 輔助資訊原則
正式規則：
- 有產品意義的 badge / 輔助資訊要保留
- 沒產品意義、只是裝飾 / 系統自述 / 流程自貼標籤的才刪

明確保留例子：
- `已建立設計交辦`
- `已建立備品交辦`
- `已建立廠商交辦`
- `尚未建立交辦`
- `共 N 筆 / N 組 / N 間`
- `#1 / #2 / #3`

### 5.5 系統自述 / helper copy
正式規則：
- 標題上下方若只是系統自述、helper copy、內部架構口吻、流程解釋，應優先移除
- 不可為了「看起來完整」重新加回多餘系統文字

### 5.6 action wording
正式規則：
- 同類動作應盡量用同一套語言
- 已固定使用方向包括：
  - `查看`
  - `查看文件`
  - `前往文件`
- 不要讓同一種動作在不同頁叫很多不同名字

## 6. UI 節奏與視覺一致性 guardrails

正式規則：
- 列表高度、卡片厚度、按鈕高度、header 對齊、section 節奏應固定
- 之後新頁 / 舊頁調整，不得隨意偏離已拍板的主線節奏

最少應遵守：
1. 列表 / 卡片高度節奏一致
2. 按鈕高度 / 大小一致
3. header 與 action 區垂直中心線一致
4. section 間距與內容節奏一致

---

## 7. 新頁 / 新模組延用規則

正式規則：
- 未來 `projectflow` 新增頁面 / 新模組 / 新支線時，必須優先沿用既有工作臺骨架與 guardrails
- 不可重新發明一套新的產品語言
- 不可脫離既有：
  - 工作臺骨架
  - `儲存 / 確認 / 文件 / 查看` 語意
  - 按鈕尺寸規則
  - header 規則
  - 空狀態語氣
  - badge / code 顯示規則
