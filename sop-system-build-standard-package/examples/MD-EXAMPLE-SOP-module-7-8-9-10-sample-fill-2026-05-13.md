# 模組 7：變更管理表（範例填寫）

## 變更項目 1
- 變更名稱：新增廠商工種多選功能
- 這次想改的是什麼：讓廠商資料頁可以選多個工種，而不是只能填單一工種。
- 這是 bug / 優化 / 新功能 / 結構調整：新功能 + 結構調整。
- 為什麼要改：原本單一工種不符合實際合作情境，一間廠商可能同時承接多種工作。
- 會影響哪些地方：廠商資料頁、建立廠商流程、資料欄位定義、可能影響搜尋與篩選。
- 改完後要重驗哪些內容：廠商建立、廠商編輯、工種顯示、篩選與搜尋。
- 這次改動要不要重開探索流程：如果只是 UI 增強可不必；若牽涉正式資料語意，就要回頭確認名詞與欄位定義。
- 這次改動會不會影響正式資料結構：會。
- 備註：這類改動不應只修前端，要確認資料來源與搜尋邏輯也同步。

## 變更項目 2
- 變更名稱：正式產品登入模式切換
- 這次想改的是什麼：讓使用者直接進系統登入頁，不再先經 Cloudflare email 驗證。
- 這是 bug / 優化 / 新功能 / 結構調整：結構調整。
- 為什麼要改：目前雙層登入不符合正式產品模式，使用者流程過長。
- 會影響哪些地方：Cloudflare Access、正式站入口、登入流程、使用者操作體驗。
- 改完後要重驗哪些內容：正式站入口、登入頁顯示、一般使用者登入、高權限路徑保護。
- 這次改動要不要重開探索流程：不一定，但要做完整盤點與風險評估。
- 這次改動會不會影響正式資料結構：通常不直接影響 DB，但會影響 auth 入口策略。
- 備註：這類改動不能只看 app，也要看站外 Access policy。

# 模組 8：階段進出場條件表（範例填寫）

## 階段 1
- 階段名稱：探索階段
- 進入這個階段前，必須先滿足什麼條件：至少有系統目標、主要使用者、基本流程想法。
- 哪些資料 / 文件不能缺：基本需求描述、至少一筆流程範例。
- 哪些驗證一定要先通過：需求方看得到前端環境。
- 這個階段結束時，應該交出什麼產物：可見流程、主要問題清單、已拍板的初步規則。
- 什麼情況下可以進下一階段：主要流程已能被看見，且需求方已指出保留與修正方向。
- 備註：這階段允許 MVP 式探索，但只限局部流程。

## 階段 2
- 階段名稱：正式化階段
- 進入這個階段前，必須先滿足什麼條件：主要流程已看過，核心語意已拍板。
- 哪些資料 / 文件不能缺：名詞定義、角色權限、流程狀態、資料需求。
- 哪些驗證一定要先通過：需求方確認目前方向正確。
- 這個階段結束時，應該交出什麼產物：正式規則、資料結構、來源定義、驗收標準。
- 什麼情況下可以進下一階段：系統邏輯已不再只停在探索討論。
- 備註：這階段的重點是把探索成果鎖進正式規則。

## 階段 3
- 階段名稱：MIM 驗收階段
- 進入這個階段前，必須先滿足什麼條件：功能已完成到可被驗收的程度。
- 哪些資料 / 文件不能缺：驗收標準、環境定義、測試資料。
- 哪些驗證一定要先通過：需求方能打開 MIM，主要流程可實際操作。
- 這個階段結束時，應該交出什麼產物：已驗過的修法、待上正式站候選清單。
- 什麼情況下可以進下一階段：修法已確定、已在 MIM 驗過、沒有重大阻塞。
- 備註：MIM 就是日常修改與驗證用測試站，不額外多切一層。

## 階段 4
- 階段名稱：正式站上線階段
- 進入這個階段前，必須先滿足什麼條件：MIM 驗過、修法已定、正式候選 commit 已整理。
- 哪些資料 / 文件不能缺：正式站候選說明、部署 / handoff 記錄。
- 哪些驗證一定要先通過：主要流程已在 MIM 驗過。
- 這個階段結束時，應該交出什麼產物：正式站可用版本、可追溯 commit 與 handoff。
- 什麼情況下可以進下一階段：正式站驗證完成。
- 備註：正式站不再拿來做探索式修改。

# 模組 9：環境分層規則表（範例填寫）

## 環境層 1
- 環境名稱：local
- 這個環境是做什麼的：執行者本機開發與初步測試。
- 允許放什麼資料：測試資料、假資料、局部驗證資料。
- 禁止做什麼：不可把本機視為需求方正式驗收環境。
- 對應的平台：本機開發環境。
- 這個環境是否對外可見：否。
- 這個環境是否可給需求方操作：通常否。
- 備註：本機只適合開發，不適合當正式驗收現場。

## 環境層 2
- 環境名稱：MIM / 測試站
- 這個環境是做什麼的：日常修改、需求確認、驗收測試。
- 允許放什麼資料：測試資料、示範資料、必要的接近真實資料。
- 禁止做什麼：不可把未確認的修法直接跳過 MIM 進正式站。
- 對應的平台：GitHub main 測試主線 + 對應測試 runtime。
- 這個環境是否對外可見：是。
- 這個環境是否可給需求方操作：是。
- 備註：這是最重要的需求確認環境。

## 環境層 3
- 環境名稱：production
- 這個環境是做什麼的：正式使用者營運與正式資料操作。
- 允許放什麼資料：正式資料。
- 禁止做什麼：不可拿來當探索場、不可隨意試修。
- 對應的平台：正式網域與正式 DB。
- 這個環境是否對外可見：是。
- 這個環境是否可給需求方操作：是，但以正式使用目的為主。
- 備註：只有在 MIM 驗過、修法已拍板後，才進 production。

# 模組 10：交接與維運責任表（範例填寫）

## 責任項目 1
- 項目名稱：正式站部署
- 主責人：系統 owner / 指定維運人
- 協作人：執行者
- 備援接手人：第二維運窗口
- 權限放在哪裡：正式部署平台帳號 / GitHub / hosting 權限
- 相關資料放在哪裡：部署手冊、handoff 文件
- 如果出問題，第一時間怎麼處理：先確認是否為 deploy 版本問題，再決定 rollback 或 hotfix。
- 備註：正式部署權限不應散落在太多人手上。

## 責任項目 2
- 項目名稱：正式站網域與入口
- 主責人：owner / 公司管理方
- 協作人：維運人員
- 備援接手人：指定備援窗口
- 權限放在哪裡：DNS / Cloudflare / 網域管理權限
- 相關資料放在哪裡：網域設定紀錄、Cloudflare 設定說明
- 如果出問題，第一時間怎麼處理：先確認 DNS / Tunnel / Access 是否正常。
- 備註：這項通常不屬於單純前端工程責任。

## 責任項目 3
- 項目名稱：資料庫與帳號
- 主責人：系統 owner / 高權限維運者
- 協作人：執行者
- 備援接手人：指定第二權限持有人
- 權限放在哪裡：Supabase / DB 管理平台 / 密鑰管理位置
- 相關資料放在哪裡：DB 文件、reset / init SOP
- 如果出問題，第一時間怎麼處理：先確認 schema、資料來源、帳號權限是否正常。
- 備註：這一項一定要有明確備援，不可只掌握在單一人手上。

## 責任項目 4
- 項目名稱：日常問題回報與接手
- 主責人：系統 owner 或指定窗口
- 協作人：執行者 / 維運者
- 備援接手人：替代窗口
- 權限放在哪裡：客服 / 回報管道 / 任務清單工具
- 相關資料放在哪裡：handoff 文件、待辦清單、變更紀錄
- 如果出問題，第一時間怎麼處理：先分類是 bug、優化還是新需求，再決定是否回 MIM 驗證。
- 備註：這項是避免所有事情都只能靠同一個人記憶處理。

## 一句話總結

> 這份範例的重點，是讓使用者看懂：系統做完後，真正成熟的不是只有功能本身，還包括怎麼管理變更、什麼時候可以進下一階段、不同環境怎麼分、以及誰負責長期維運與接手。
