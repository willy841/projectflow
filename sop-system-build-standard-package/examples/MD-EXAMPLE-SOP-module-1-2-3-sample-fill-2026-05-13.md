# MD-EXAMPLE — SOP 模組 1～3 範例填寫版 — 2026-05-13

Status: draft
Purpose: 示範 SOP 模組 1～3（名詞、角色、流程）一份比較好的填法，讓一般使用者更直觀看懂題目在問什麼、答案應該長什麼樣。

Important:
- 這份是**示範用**，不是正式規格。
- 內容是用「假想專案管理 / 接案流程系統」當例子。
- 真正新專案時，應依實際需求重填。

---

# 模組 1：名詞定義表（範例填寫）

## 名詞 1
- 名詞名稱：專案
- 白話解釋：一筆完整案件，從開始接觸、報價、執行到結束，都算同一個專案。
- 這個詞通常出現在什麼情境：建立新案、查詢進度、查看總成本、結案。
- 容易和哪個詞混淆：任務
- 和那個詞差在哪裡：專案是整筆案件；任務是專案底下的一個執行項目。
- 這是不是正式狀態 / 正式資料：是，屬於正式主資料。
- 備註：每個專案可以有很多任務。

## 名詞 2
- 名詞名稱：任務
- 白話解釋：專案底下實際要做的事情，例如設計、採購、施工安排。
- 這個詞通常出現在什麼情境：派工、追蹤進度、分配執行人員。
- 容易和哪個詞混淆：專案
- 和那個詞差在哪裡：任務只是專案的一部分，不是整筆案件。
- 這是不是正式狀態 / 正式資料：是，但屬於從屬資料。
- 備註：一個專案可以拆成多個任務。

## 名詞 3
- 名詞名稱：正式確認
- 白話解釋：某個步驟已被允許進入正式狀態，不再只是暫時草稿。
- 這個詞通常出現在什麼情境：確認需求、確認報價、確認發包內容。
- 容易和哪個詞混淆：暫存
- 和那個詞差在哪裡：暫存只是先記下來；正式確認表示可以往下一步走，而且通常會影響其他資料。
- 這是不是正式狀態 / 正式資料：是。
- 備註：正式確認後，通常修改權限會收緊。

## 名詞 4
- 名詞名稱：結案
- 白話解釋：整個專案已經完成，不再繼續新增主要工作內容。
- 這個詞通常出現在什麼情境：專案收尾、財務結算、歷史查詢。
- 容易和哪個詞混淆：完成
- 和那個詞差在哪裡：完成可能只是某一步做完；結案是整筆專案正式收尾。
- 這是不是正式狀態 / 正式資料：是。
- 備註：結案後若要再改，通常要走重開流程。

## 名詞 5
- 名詞名稱：廠商
- 白話解釋：系統中可被指派、合作、付款或對帳的外部合作單位。
- 這個詞通常出現在什麼情境：任務發包、報價承接、付款與對帳。
- 容易和哪個詞混淆：供應商
- 和那個詞差在哪裡：若系統裡兩者其實同義，就應直接統一只留一種說法；不要混用。
- 這是不是正式狀態 / 正式資料：是。
- 備註：若未來確定供應商與廠商無差別，系統內就只保留「廠商」。

# 模組 2：角色與權限表（範例填寫）

## 角色 1
- 角色名稱：管理者
- 這個角色平常負責什麼：管理整體系統、建立使用者、設定主要規則、查看所有資料。
- 可查看哪些內容：全部專案、全部任務、全部廠商、全部財務資料。
- 可新增哪些內容：專案、任務、廠商、使用者。
- 可修改哪些內容：大部分內容都可修改。
- 可刪除哪些內容：可刪除錯建資料，但高風險刪除應謹慎處理。
- 可確認 / 核准哪些動作：正式確認、結案、重開、權限調整。
- 不可操作哪些項目：不應直接跳過記錄流程做隱性修改。
- 備註：通常是系統 owner 或高權限主管。

## 角色 2
- 角色名稱：專案執行者
- 這個角色平常負責什麼：建立專案內容、更新進度、整理需求、推進流程。
- 可查看哪些內容：自己負責的專案與相關任務。
- 可新增哪些內容：任務、備註、進度更新。
- 可修改哪些內容：自己負責專案中的未確認資料。
- 可刪除哪些內容：通常不可直接刪正式資料。
- 可確認 / 核准哪些動作：可送出確認，但不一定有最終核准權。
- 不可操作哪些項目：不可任意調整高權限設定與他人帳號。
- 備註：是主要日常使用者之一。

## 角色 3
- 角色名稱：主管 / 核准者
- 這個角色平常負責什麼：審核重要節點，決定是否正式確認、是否進下一階段。
- 可查看哪些內容：被指派審核的專案與相關摘要。
- 可新增哪些內容：通常不負責細部建立。
- 可修改哪些內容：可改審核結果、退回意見。
- 可刪除哪些內容：通常不可直接刪除底層資料。
- 可確認 / 核准哪些動作：正式確認、核准報價、核准結案。
- 不可操作哪些項目：不應直接代替執行者做大量基礎輸入。
- 備註：此角色重點在拍板，不在日常細部維護。

## 角色 4
- 角色名稱：外部廠商
- 這個角色平常負責什麼：查看自己被分配到的內容，回覆資料，提交執行結果或文件。
- 可查看哪些內容：只看與自己相關的專案 / 任務 / 文件。
- 可新增哪些內容：可上傳回覆、文件、必要資料。
- 可修改哪些內容：只可修改自己提交但尚未確認的內容。
- 可刪除哪些內容：通常不可刪除正式紀錄。
- 可確認 / 核准哪些動作：通常沒有系統內正式核准權。
- 不可操作哪些項目：不可看全部專案、不可看他人資料、不可改系統設定。
- 備註：若未來外部廠商不直接登入，這角色可先保留為規劃角色。

# 模組 3：流程與狀態表（範例填寫）

## 流程名稱
- 流程名稱：專案建立到正式確認流程
- 這個流程是做什麼的：把一筆新需求從建立、整理、確認，推進到可正式開始執行的狀態。
- 這個流程的起點是什麼：收到新需求。
- 這個流程的終點是什麼：專案正式確認，可往下進入執行與派工。

---

## 步驟 1
- 目前步驟名稱：建立新專案
- 誰負責這一步：專案執行者
- 進入這一步的條件：有新的案件 / 新的合作需求。
- 使用者在這一步要做什麼：填寫專案基本資料、名稱、需求摘要。
- 完成後下一步是什麼：進入需求整理。
- 這一步可不可以修改：可以。
- 這一步是不是正式狀態：不是，屬於初始建立。
- 如果這一步做錯了，要怎麼處理：可直接修改，必要時可作廢後重建。
- 備註：這一步重點是先建立案件存在。

## 步驟 2
- 目前步驟名稱：整理需求與內容
- 誰負責這一步：專案執行者
- 進入這一步的條件：專案已建立。
- 使用者在這一步要做什麼：補齊需求、拆任務、整理必要資料。
- 完成後下一步是什麼：送交主管審核。
- 這一步可不可以修改：可以。
- 這一步是不是正式狀態：不是，仍屬整理中。
- 如果這一步做錯了，要怎麼處理：持續修正，直到內容足夠完整。
- 備註：這一步通常是最容易反覆調整的階段。

## 步驟 3
- 目前步驟名稱：主管審核
- 誰負責這一步：主管 / 核准者
- 進入這一步的條件：需求內容已整理完成。
- 使用者在這一步要做什麼：確認資訊是否足夠、流程是否合理、是否可往下走。
- 完成後下一步是什麼：若通過，進入正式確認；若不通過，退回需求整理。
- 這一步可不可以修改：主管通常不改細部內容，而是退回要求修正。
- 這一步是不是正式狀態：還不是，這是確認前最後審核。
- 如果這一步做錯了，要怎麼處理：可退回上一階段重新整理後再審。
- 備註：這一步重點是拍板，而不是代做。

## 步驟 4
- 目前步驟名稱：正式確認
- 誰負責這一步：主管 / 管理者
- 進入這一步的條件：審核通過。
- 使用者在這一步要做什麼：將專案標記為正式確認，允許後續執行。
- 完成後下一步是什麼：進入執行 / 派工流程。
- 這一步可不可以修改：通常不能隨意改，若要改要走退回或重開流程。
- 這一步是不是正式狀態：是。
- 如果這一步做錯了，要怎麼處理：需由高權限角色執行退回或重開。
- 備註：這一步是流程的重要分界點。

## 例外情況補充

### 例外情況 1
- 這是什麼情況：需求整理到一半發現案件取消。
- 會在哪一步發生：步驟 1 或步驟 2。
- 發生後要怎麼處理：將專案標記為取消 / 作廢，不再往下走。
- 誰有權限處理：專案執行者可提報，管理者或主管可正式確認取消。
- 處理完會回到哪個狀態：不回主流程，改進入取消狀態。
- 備註：取消案件應保留紀錄，不建議直接硬刪除。

### 例外情況 2
- 這是什麼情況：正式確認後發現關鍵資訊錯誤，需要重開。
- 會在哪一步發生：步驟 4 之後。
- 發生後要怎麼處理：由高權限角色啟動重開流程，回到需求整理或審核階段。
- 誰有權限處理：主管 / 管理者。
- 處理完會回到哪個狀態：回到整理中或待審核。
- 備註：這是高風險例外流程，應明確留紀錄。

## 一句話總結

> 這份範例的目的，不是告訴你固定答案，而是讓你直觀看到：一份好的 SOP 填寫結果，不是只寫幾個名詞或一句話，而是要把名詞語意、角色責任、流程步驟與例外情境都講到足以讓別人接手。
