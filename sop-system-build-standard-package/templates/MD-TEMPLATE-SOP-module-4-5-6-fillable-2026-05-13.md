# 模組 4：資料蒐集清單（可直接填寫版）

## 這份表要解決什麼

很多專案不是不能做，而是一開始不知道到底要準備哪些資料。

結果就會變成：
- 做到一半才補資料
- 重要欄位忘了講
- 需求方以為已經講清楚，其實還缺很多關鍵資訊
- 執行者要一直回頭追問

這份表的目的，是把「開工前要先準備哪些資料」列清楚。

---

## 填寫方式

每一列填一種資料。

你不需要一開始就知道很技術的欄位名稱，
只要先把「這筆資料是什麼、為什麼需要、缺了會怎樣」講清楚。

# 模組 5：環境與帳號清單（可直接填寫版）

## 這份表要解決什麼

這份表是要先講清楚：
- 使用者在哪裡看前端
- 開發 / 測試 / 正式站分別在哪裡
- 需要哪些帳號與權限
- 誰負責提供 GitHub / Vercel / Supabase 之類的環境資訊

如果這個不先定，後面很容易：
- 站做出來但需求方看不到
- 帳號缺權限無法操作
- 測試站與正式站混在一起
- DB、前端、部署責任不清楚

---

每一列填一個環境。

建議至少分：
- local（本機）

# 模組 6：驗收標準表（可直接填寫版）

## 這份表要解決什麼

這份表是要先講清楚：
- 怎樣才算完成
- 怎樣不算完成
- 驗收要看什麼
- 誰來驗
- 驗收時要拿出什麼證據

如果沒有這份表，後面最容易發生：
- 做完了但雙方對「完成」理解不同
- 使用者覺得還不行，執行者覺得已做完
- 只看畫面，沒驗資料
- 只驗單頁，沒驗整體一致性

---

## 填寫方式

每一列填一種驗收項目。

建議至少分成：
- 功能驗收
- 操作驗收
- 資料驗收
- 環境驗收

## 建議驗收分類

### 功能驗收
- 這功能有沒有真的做出來

### 操作驗收
- 使用者實際操作時順不順

### 資料驗收
- 資料有沒有寫對、同步對、顯示對

### 環境驗收
- 測試站 / 正式站 / 登入 / 部署有沒有正常

---

## 填寫提醒

如果你不知道怎麼寫成功條件，就不要寫：
- 看起來正常
- 大致可用
- 差不多完成

要改成具體句子，例如：
- 可以成功登入
- 可以建立資料
- 建立後列表與明細都看得到
- 重新整理後資料不會消失
- 一般使用者不能看到管理功能

---

## 一句話總結

> 這份模板的目的，是把 SOP 的中段核心——資料、環境、驗收——從概念整理成可直接填寫的格式，讓後續不只是知道要做什麼，也知道要準備什麼、在哪裡做、以及怎樣才算真的完成。
