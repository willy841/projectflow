# MD107 — projectflow home overview usage scenario alignment summary (2026-04-13)

> Status: ACTIVE
> Note: 本文件用來整理目前已核對過的 `projectflow` 首頁總覽（Home Overview）使用情境、功能責任與資料承接判斷，供後續逐區盤點與回查使用。


## 1. 文件定位

本文件用來把目前已核對過的首頁總覽使用情境，整理成一份結構化摘要。

本文件目的：
- 固定目前對首頁總覽的產品共識
- 區分哪些首頁數字應承接 quote-cost 主線，哪些之後應由 Accounting Center 提供
- 作為後續繼續盤點帳務中心與首頁對接關係時的基準文件

---

## 2. Header 區塊

### 使用情境
首頁 header 應顯示：
- 系統名稱：`酷亞專案系統`
- 快速提示：
  1. 總共有多少個專案
  2. 多少筆交辦未處理

### 明確拿掉的內容
- `本月預估毛利`
- `查看專案列表`

### 保留的主操作
- `新增專案`
  - 作為建立專案的入口

### 判斷
- 功能邏輯：正確
- 首頁作為系統總覽 + 專案建立入口，與整個 projectflow 上游架構吻合
- `本月預估毛利` 會搶走 financial 主線的定義權，移除是正確的
- `查看專案列表` 屬冗餘 CTA，可移除

### 後續值得補定義
- `多少筆交辦未處理` 的計算口徑
- `總共有多少個專案` 是否只算 active projects 或全部 projects

---

## 3. 營運摘要指標區塊

### 使用情境
首頁下一區塊應顯示四個數字：
1. `進行中的專案數量`
2. `待處理設計的任務數量`
3. `待採購備品的任務數量`
4. `待發包廠商的任務數量`

### 待處理的正式定義
> 只要任務發布後，還沒有按 `最終確認` 的，都叫做 `待處理`。

也就是：
- 設計 / 備品 / 廠商三條線的首頁待處理數量
- 其共同計算口徑為：
  - `已發布`
  - 但 `尚未最終確認`

### 判斷
- 功能邏輯：正確
- 這比單一模糊的未處理交辦數更有管理價值
- 與三條執行線（設計 / 備品 / 廠商）的流程與狀態邏輯吻合

### 後續值得補驗
- `進行中的專案數量` 是否以尚未結案的 active project 為準
- 三條線首頁待處理數量是否 truly 以 `發布後但尚未最終確認` 為計算口徑落地

## 4. 近期專案區塊

### 使用情境
首頁往下需要有一個 `近期專案` 區塊，用來顯示近期專案的列表。

### 判斷
- 功能邏輯：正確
- 與首頁作為 overview page 的角色吻合
- 作為最近動態的專案入口合理

## 5. 本月財務摘要

### 使用情境
首頁還需要有一個 `本月財務摘要` 區塊，應包含：
1. `本月專案總額`
2. `已收款`
3. `未收款`

---

### 5.1 本月專案總額

#### 定義
- 指在這個月份之中
- 我們執行中或已完成的專案
- 只要是在這個月份 `對外報價的總額` 都會計入

#### 來源邊界
- 這個資料來源應來自 `帳務中心`
- 目前先 `不做正式對接`

#### 判斷
- 功能邏輯：正確
- 屬於月份口徑的 financial total
- 應先註記來源 ownership，等帳務中心盤完再正式接入

---

### 5.2 已收款 / 未收款

#### 定義
- `已收款`
  - 指我們在報價成本中，已經標記收了多少錢
- `未收款`
  - 指將總額扣除已收款後的餘額

#### 重要補充
- 這兩個數字是 `執行中的加總`
- **跟月份無關**

#### 判斷
- 功能邏輯：正確
- 應直接來自報價成本線的 active aggregation
- 與 quote-cost / 收款管理主線完全吻合

---

### 5.3 統計口徑提醒

這三個數字雖然可以放在同一區塊，但其統計口徑不同：
- `本月專案總額`
  - 月份口徑
  - 來源應屬帳務中心
- `已收款 / 未收款`
  - active project aggregation
  - 來源應屬報價成本 / 收款管理
  - 與月份無關

#### 判斷
- 同區塊顯示是可以的
- 但 UI 文案與資料來源邊界必須標清楚，避免被誤認為三者都是月度統計

---

## 6. 首頁總覽總結

### 目前已清楚且與正式架構方向吻合的部分
1. Header 應顯示系統名稱 `酷亞專案系統`
2. 首頁主 CTA 只保留 `新增專案`
3. `本月預估毛利` 與 `查看專案列表` 可移除
4. 首頁需要營運摘要指標：進行中專案數量 + 三條執行線待處理數量
5. `待處理` 的正式定義是：任務已發布，但尚未按 `最終確認`
6. 首頁需要 `近期專案` 區塊
7. 首頁需要 `本月財務摘要` 區塊
8. `已收款 / 未收款` 應來自報價成本主線，且屬 active aggregation

### 目前已定義方向，但需後續正式對接 / 補定義的部分
1. `總共有多少個專案` 的計算口徑
2. `多少筆交辦未處理` 是否仍保留，或是否被下方四個指標充分取代
3. `本月專案總額` 的正式來源應屬帳務中心，目前先不對接
4. 同一區塊內月份口徑與 active aggregation 口徑的文案標示需明確

---

## 7. 一句話總結

> 目前已核對的 `projectflow` 首頁總覽使用情境整體是清楚且與正式架構方向吻合的：首頁應作為系統總覽與新增專案入口，透過 header、營運摘要指標、近期專案與本月財務摘要四個區塊，提供上游建立專案、三條執行線待處理數量，以及 financial overview 的高層視圖；其中 `已收款 / 未收款` 應直接承接報價成本主線，而 `本月專案總額` 之後應屬帳務中心來源，目前先註記、不正式對接。
