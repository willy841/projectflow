## C. 目前 vendor flow 討論到的核心結論

### 1. 單筆廠商交辦，不等於對外正式發包單
目前已確認的方向是：

- 單筆廠商交辦 = 內部逐項管理單位
- 真正對外送廠商的，應是「同一廠商的彙整發包單」

也就是：
- 內部逐項建立
- 對外依同廠商彙整

---

### 2. 廠商交辦應分兩層

#### 第一層：Vendor Assignment（逐項）
用途：
- 掛在 execution item 底下
- 管理單一項目的需求、規格、預算、備註、狀態、回覆

#### 第二層：Vendor Package / Vendor Order Sheet（彙整）
用途：
- 同一專案、同一廠商的多筆項目整併
- 生成對外可複製 / 可匯出的發包內容
- 作為正式發包確認的主單位

---

### 3. 先前已釐清的回覆邏輯
這輪已拍板：

- 回覆區不做樹狀結構
- 回覆是單層平面列表
- 新增回覆為 append 新的一筆
- 不做子回覆 / 討論串視覺暗示

此邏輯目前主要用於 assignment reply 機制。

---

### 4. 金額確認的真正語意
你已明確指出：

- 在廠商回覆頁面裡的「金額確認」
- 本質上代表：**已正式發包**

因此未來命名與流程不應只寫成中性「金額確認」，
而應視為發包成立的正式動作。

但此動作究竟掛在：
- 單筆 vendor assignment
- 或 vendor package（彙整單）

目前傾向後者，尚待最終拍板。

---

## D. 目前傾向中的產品方向

目前討論後，較合理的產品方向是：

1. 先在 execution item 底下建立多筆 vendor assignment
2. 每筆 assignment 屬於內部逐項管理單位
3. 系統應能依「同專案 + 同廠商」自動歸組
4. 產生 vendor package / 發包彙整單
5. 對外真正要送廠商的是彙整單，不是逐筆 assignment
6. 正式發包確認，很可能應掛在彙整單層，而非單筆層

---

## E. 尚待你拍板的 4 個問題

### 1. 彙整單的歸組單位
是否確定為：
- **同專案 + 同廠商**

也就是不同專案不可混成同一張發包單。

### 2. 單筆 vendor assignment 是否必須先指定廠商
選項可能是：
- 建立時就指定單一主廠商
- 可先不指定，之後再歸組
- 可先列候選廠商，最後再定主廠商

### 3. 廠商回覆掛在哪一層
選項可能是：
- 以單筆 assignment 為主
- 以 vendor package 為主
- 兩層都可，但語意不同

### 4. 正式發包確認掛在哪一層
選項可能是：
- 單筆 assignment
- vendor package（較傾向）
