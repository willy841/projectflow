# 2. 使用者合作偏好與工作原則

## 2.1 溝通與驗收偏好
- 全程使用 **繁體中文**
- 使用者偏好 **先做看得到、可立即驗收的前端修改**
- UI 驗收很細，特別在意：
  - 按鈕可讀性
  - badge 對齊
  - 桌機 100% 顯示比例排版
  - 畫布寬度 / 側欄寬度 / 表格寬度
  - 命名與資訊架構是否一致
- 不要只講抽象規劃，要能直接落成畫面或互動

### 2.1-A UI 工作原則（2026-04-04 CPO 收斂版）
本段整理使用者對 `projectflow` UI / UX 的穩定要求。後續若要續接 UI 微調、驗收、派工或做 handoff，應優先以本段作為高階準則。

#### A. UI 的核心目標
使用者要的不是「設計稿式漂亮畫面」，而是：
> **高資訊密度、可快速判讀、可立即操作、桌機 100% 比例下穩定、主次清楚的管理介面。**

也就是：
- 可以密，但不能亂
- 可以多，但不能雜
- 可以快修，但不能失去系統性

#### B. 驗收時最在意的 UI 指標
後續驗收時，特別優先看：
1. 按鈕可讀性
2. badge 對齊與垂直置中
3. 桌機 100% 顯示比例排版穩定度
4. 畫布寬度 / 側欄寬度 / 表格寬度是否合理
5. 命名、狀態語言、資訊架構是否一致
6. 首屏是否能快速判讀，不靠長段 helper copy 補救

#### C. 使用者偏好的 UI 方向
1. **高資訊密度，但主次一定要清楚**
   - 可接受管理台型 UI
   - 但首屏不能塞到失去辨識力
   - 展開後視線要自然往下走

2. **主線資訊要優先，背景 / fallback / 例外資訊要降階**
   - 不接受 seed / fallback / 補充資訊搶主線注意力
   - 成本主線、文件主線、任務主線都要先被看懂

3. **CTA 主次要非常明確**
   - 主按鈕要一眼看出來
   - 輔助操作不能搶走主流程
   - `查看整理內容`、`查看文件`、`生成文件` 等操作要有固定節奏

4. **狀態語言與視覺要像同一套系統**
   - badge 名稱要一致
   - 文件狀態、回覆狀態、操作狀態要用同一語氣
   - 顏色與命名不能在不同頁面互相打架

5. **對齊不是裝飾，而是品質本體**
   - 表頭與搜尋列對齊
   - label 基線對齊
   - badge 垂直置中
   - 按鈕高度一致
   - 卡片 / 區塊左右基準線一致

#### D. 明確不喜歡的 UI 問題
1. 多餘說明文 / helper copy 太多
2. 同一頁看起來像三套 UI 拼起來
3. 靠零碎 padding / margin 補位硬修版型
4. 主線資訊被例外資訊、fallback 資訊、背景資訊搶焦點

#### E. 後續派工時應直接遵守的 8 條 UI 準則
1. 高資訊密度可以，但主次一定要清楚
2. 桌機 100% 比例下，排版不能跑掉
3. 按鈕、badge、label、欄位基線必須對齊
4. 主 CTA 要明確，輔助操作不能搶主流程
5. 狀態語言、顏色、命名必須跨頁一致
6. 首屏只留判讀必要資訊，不要塞滿
7. 主線資訊優先；fallback、例外、背景資訊降階
8. 不要靠零碎 padding 補丁修版型；版型問題要從結構解

## 2.2 協作規則
- 當 context 接近飽和，必須主動提醒、收口並整理交接 MD
- 當 usage 剩餘約 10%，也必須執行相同收口流程

# 5. Vendor Flow：目前正確的產品結構

這一段是 `projectflow` 現階段最重要的新能力主線之一。

## 5.1 Vendor Flow 已定為三層結構

### Layer 1：Assignment / Reply
定位：
- 內部逐項管理單位
- 在這一層整理單項內容、回覆、vendor 選擇
- 使用者在 reply 階段正式選定 vendor 並按下 **發包**

產品語意：
- 這是 **發包前（pre-issue）** 的決策區
- assignment 不等於對外正式發包單

### Layer 2：Vendor Package
定位：
- 同專案 + 同廠商的唯一主容器
- 作為對外需求整理與確認主線
- package 的主要責任不是終點，而是整包整理 / 編輯前台

產品語意：
- package 屬於 **發包後（post-issue）** 主線
- package 不是最終文件，而是文件生成前台

### Layer 3：Final Outgoing Document
定位：
- package 整理完成後，手動生成的最終對外文件
- 給廠商執行使用
- 不是報價單，不顯示金額

產品語意：
- 這份文件才是最終對外成果
- 不是 package 的即時畫面投影，而是當前內容輸出的正式結果

## 5.2 智慧歸包規則（已拍板）
規則：
1. 使用者在 assignment / reply 層選定 vendor
2. 按下 `發包`
3. 系統依 **同專案 + 同廠商** 規則自動歸包
4. 若該 project 下該 vendor 已有 package，assignment 自動進入該 package
5. 若沒有，系統自動建立新的 package

重要原則：
- **同專案 + 同廠商不應存在多個 package**
- 也就是：
  - `Project + Vendor = 唯一 package 主體`

## 5.3 已排除的錯誤方向
以下方向已被正式推翻：
- 不把單筆 vendor assignment 直接當對外正式發包單
- 不把 package 當最終終點
- 不把任務流中出現過的 vendor，直接視為正式 Vendor Flow 主線
- 不把所有 vendor 資訊混成單一區塊呈現

# 6. Project Detail 中的 Vendor 正確資訊架構

## 6.1 最重要的產品修正
在 task / assignment 流程中出現的 vendor，**不等於正式發包 vendor**。

只有當：
- 在 reply 階段正式選定 vendor
- 並按下 `發包`

才進入 package 主線。

## 6.2 Project Detail 必須拆成兩區
已拍板：

### A. Assignment / Reply 區
- 屬於 **pre-issue** 區
- 處理正式發包前的事情
- 包含：
  - assignment / reply 流程
  - vendor 決策
  - 發包入口

建議欄位：
- Assignment 名稱
- Reply 狀況
- 已選定 Vendor
- 發包狀態
- 操作（查看 Reply、發包）

### B. Issued Packages 區
- 屬於 **post-issue** 區
- 只處理已正式發包後的 package 主線

建議欄位：
- 廠商名稱
- assignment 數
- 文件狀態
- 操作（查看 Package）

文件狀態名稱：
- 未處理
- 已處理
- 需更新

## 6.3 區塊順序
由上到下：
1. Assignment / Reply
2. Issued Packages

也就是：

> Pre-Issue → Post-Issue

# 7. Final Outgoing Document：已定規則

## 7.1 文件定位
Final Outgoing Document 是：
- 對外給廠商執行用的最終文件
- 不是內部管理文件
- 不是報價單
- **不顯示金額**

## 7.2 文件欄位結構（已定稿）

### A. 專案資訊區
欄位：
- 專案名稱
- 客戶名稱
- 活動日期
- 進場時間
- 地點

規則：
- 預設從 project 自動帶入
- package 可局部覆寫
- 覆寫值屬於 package，不回寫 project 主資料

### B. 發包項目明細區
每列欄位：
- 項次
- 項目名稱
- 工作內容 / 規格說明
- 數量
- 單位
- 備註

規則：
- 由 package 層人工整理
- 不直接等於 assignment 原始資料
- 需支援：新增 / 編輯 / 刪除 / 排序

### C. 文件整體備註區
欄位：
- 整體備註

規則：
- 由 package 維護
- 文件生成時一併輸出

## 7.3 文件生成規則
- 使用者在 package 中整理完成後，手動按下 `生成發包文件`
- 若 package 內容之後被修改，系統需提示：
  - **目前發包文件不是最新內容，請重新生成**
- 使用者按 `重新生成發包文件` 時，系統需先要求確認

## 7.4 單一有效文件規則
已拍板：
- **同一 package 同時間只保留一份有效文件**
- 不保留舊版本歷史
- 重新生成時，以新文件覆蓋舊文件

# 11. Vercel 部署關鍵設定

## 11.1 Root Directory
先前已確認過的重要設定：
- Vercel Root Directory **不能是 `/`**
- 應設為：

```txt
project-mgmt
```

若設錯，會出現：
- `Couldn't find any 'pages' or 'app' directory. Please create one under the project root`

## 11.2 Framework detection
先前曾發生：
- Deployment 顯示 Ready / Production / Current
- 但實際 routes 全站 404
- Deployment Summary 顯示 `No framework detected`

為了降低這種風險，已補：
- `project-mgmt/vercel.json`

內容：
```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "framework": "nextjs"
}
```

## 11.3 Production branch 心智
已知重要背景：
- Vercel Production branch 曾實際吃 `main`
- 不可主觀假設是 `master`
- 若 GitHub 修改在 `master`，但 Vercel Production 看 `main`，就會出現線上內容落差

因此部署相關判斷要遵守：
1. 先確認 GitHub push 到哪個 branch
2. 再確認 Vercel Production 看哪個 branch
3. 最後才判斷線上是否應該已更新

## 11.4 部署前檢查 SOP
在你準備把某次修改送上線前，至少確認：
- 你 push 的 branch 是什麼
- Vercel Production branch 是什麼
- Root Directory 是否為 `project-mgmt`
- repo 內是否有 `project-mgmt/vercel.json`

## 11.5 部署後驗收重點
部署後至少驗：
- `/`
- `/projects`
- `/projects/<project-id>`
- vendor 相關路由

避免出現：
- 只剩靜態資產可訪問
- routes 404
- Ready 但實際不可用

## 11.6 建議的部署後驗收 SOP
每次 deploy 後，至少做以下驗收：

### Step 1：確認 deployment 狀態
看 Vercel 上是否為：
- Ready
- Production
- Current

### Step 2：打開首頁與核心路由
至少檢查：
- `/`
- `/projects`
- 一個專案 detail 頁
- 一個 vendor package / vendor assignment 相關頁

### Step 3：確認不是只有靜態資產活著
如果你看到：
- `/next.svg`
- `/vercel.svg`
- `/globe.svg`
這類靜態檔可開，但 app routes 全 404，代表 deployment 雖然 Ready，serving 仍可能有問題。

### Step 4：確認這次上線內容真的存在
至少驗一個這輪修改最明確的畫面或文案，不要只看 deployment 狀態。

## 11.7 看到這些症狀時，優先懷疑什麼
### 症狀 A：build 成功但 routes 404
優先懷疑：
- Root Directory 錯
- framework detection 異常
- Vercel 沒把它當 Next.js app 正常 serving

### 症狀 B：GitHub 有新 commit，但線上沒變
優先懷疑：
- push 到錯 branch
- Vercel Production branch 看的是另一個 branch
- 最新 deploy 沒吃到正確 commit

### 症狀 C：module-not-found
優先懷疑：
- 依賴檔沒一起 push 上 GitHub
- 只推了局部修改，支撐檔缺失

## 11.8 部署判斷的禁止事項
- 禁止只因為 deployment 顯示 Ready 就判定沒問題
- 禁止只因為 GitHub 有 commit 就判定已上線
- 禁止沒驗 routes 就說 deploy 完成
- 禁止忽略 branch / root directory / framework detection 這三個核心因素

最新定位：

- 已送出不可再次送出，也不可重複建立同 assignment 的 package item

- 每筆項目的項目名稱（可編輯）
- 每筆項目的需求說明（可編輯）
- 文件整體備註
- 主操作按鈕：`生成文件` / `重新生成文件`

#### 文件背景欄位
- 專案名稱
- 活動日期
- 地點
- 進場時間

規則：
- 以上背景值在 package 建立時固定由 project detail 主資料帶入
- 帶入後屬於 package 自己的副本
- package 可局部調整，但不回寫 project 主資料

#### 單筆項目列欄位
- 編號
- 項目名稱（可編輯）
- 需求說明（可編輯）

規則：
- package item 只承接：項目名稱、需求說明
- 工種不進 package 主欄位，因為 package 已經依廠商歸包
- 工種、預算、參考連結、回覆、vendor 選擇與其他任務管理資訊都不進 package 主欄位

#### Package 層核心任務
不是選 vendor，而是：

> **整理每一筆真正要對外發出的需求內容，並直接生成文件**

#### Package 層不需要再額外膨脹的功能
- 不要過多欄位
- 不要過多中間狀態
- 不要過多複雜操作
- 不要另一套回覆系統
- 不要另一套發包判斷流程
- 不要太重的管理表單

### C. 最終文件（Final Outgoing Document）最新定稿
最終文件不是複雜表格，也不是報價單格式，而是：

> **可直接傳給廠商閱讀與執行的條列式文本**

#### 固定結構
1. 第一行：日期 + 活動名稱
2. 第二行：地點
3. 第三行：進場時間
4. 第四段：需求內容（固定標題）
5. 最後：備註區

#### 規則
- 第一行固定為：日期 + 空格 + 活動名稱
- 第二行固定為：`地點：` + 地點內容
- 第三行固定為：`進場時間：` + 時間
- `需求內容` 標題固定叫「需求內容」
- 底下項目一定要有編號
- 每筆需求內容固定格式為：`項次. 項目名稱：需求說明`
- 文件最後保留備註區
- 若無備註內容，固定顯示 `-`
- 文件目標是：清楚、好讀、可直接傳給廠商執行

#### 範例
```text
2026/04/12 春季品牌快閃活動
地點：松山文創園區 2F 主展區
進場時間：09:00

需求內容
1. 入口主背板輸出與安裝：主背板輸出、裱板、現場安裝與收邊
2. 導視系統與桌牌輸出：入口指示、動線牌與桌牌輸出

備註
如現場尺寸有變動請提前告知
```

- 作為對外輸出成果

## 16.6-A 設計線第一版定稿（2026-04-03 夜間更新版）

> **設計回覆不是自動全部進文件主線；只有被正式確認的回覆，才可進入設計文件整理與設計文件。**

### A. 設計任務卡 / 展開層

第一屏不顯示：

- 負責人

- 執行廠商

正式語意：
- 第一屏是 **追蹤摘要卡 / 入口卡**
- 不是總表卡、報表卡或資訊全塞卡

#### 展開層結構
展開後應分成兩層：
1. **母卡層：交辦設計**
2. **回覆層：R1 / R2 / R3**

##### 母卡層：交辦設計
角色：
- 原始設計需求層
- 顯示任務母卡 / 初始 spec

應顯示：
- 項目
- 尺寸
- 材質 + 結構
- 數量
- 需求說明
- 參考連結
- 負責人
- 狀態

##### 回覆層：R1 / R2 / R3
角色：
- 逐次設計提案 / 修正版資料卡
- 不做樹狀
- 不是 chat bubble，也不是 comment thread

每筆正式欄位：
- 回覆標題
- 數量
- 金額
- 尺寸
- 材質 + 結構
- 檔案位置（URL）
- 執行廠商

正式規則：
- 每張回覆卡要明確標示 `R1 / R2 / R3 / ...`
- 回覆卡閱讀順序為：
  1. R 序號
  2. 回覆標題
  3. 執行廠商
  4. 尺寸 / 材質 + 結構 / 數量
  5. 檔案位置（URL）
  6. 金額
- 金額是正式欄位，但不是第一閱讀焦點
- 執行廠商權重高於金額，因為後續文件整理與成本歸屬都會承接此欄位

#### 母卡與回覆層關係
正式拍板：
- 母卡層是原始需求
- 回覆層是逐次提案
- 兩者必須清楚分層，不可做成同一種卡

### B. 回覆確認規則（2026-04-03 夜間新增）
本段為今日最新補拍板的重要新規則，後續若與較早期描述衝突，以本段為準。

#### 回覆層兩種狀態
每筆設計回覆在回覆層必須區分：
1. **未確認回覆**
2. **已確認回覆**

##### 未確認回覆
定義：
- 已有回覆內容
- 但尚未完成正式確認
- 只能留在回覆層
- **不得進入設計文件整理層**
- **不得計入成本**

##### 已確認回覆
定義：
- 已完成正式確認
- 可進入設計文件整理層
- 可成為設計文件內容來源
- 可成立正式成本

#### 正式確認條件
一筆設計回覆要成為 **已確認回覆**，必須同時滿足：
1. **已選定執行廠商**
2. **已按儲存**

正式語意：

> **在設計回覆層，選了執行廠商並按儲存，不是單純存資料，而是正式確認該筆回覆進入文件主線。**

### C. 設計文件整理層（夜間更新版）
定位：

> **依執行廠商分組的「已確認回覆整理入口層」**

核心規則（更新後）：
- **只吃已確認回覆**
- 不吃未確認回覆
- 不只看最新一筆
- 不做版本覆蓋
- 依執行廠商分組

正式語意：
- 一個執行廠商 = 一組設計文件整理單位
- 一個執行廠商 = 一份設計文件
- 整理層不是回覆總表，而是已確認內容的整理入口

整理層每個分組卡的主閱讀順序：
1. 執行廠商
2. 文件狀態
3. 已確認回覆數 / 項目數
4. 查看整理內容

每個執行廠商分組卡正式保留：
- 執行廠商名稱
- 文件狀態
- 已確認回覆數 / 項目數
- 主操作：`查看整理內容`
- 條件型次操作：`查看文件` / `生成文件` / `重新生成文件`

正式拍板：
- `查看整理內容` 是固定主操作
- `查看文件` 不是主操作
- 未確認回覆不得出現在設計文件整理層

#### 文件狀態與操作對應
- `未生成`
  - 代表該執行廠商底下已有已確認內容，但尚未生成正式文件
  - 主操作：`查看整理內容`
  - 次操作：`生成文件`
- `已生成`
  - 代表目前文件為最新版本
  - 主操作：`查看整理內容`
  - 次操作：`查看文件`
- `需更新`
  - 代表已確認內容有變更，現有文件不再是最新
  - 主操作：`查看整理內容`
  - 次操作：`查看文件` + `重新生成文件`

#### 何時進入 `需更新`
- 某筆已確認回覆被修改
- 或新的回覆被確認進來

### D. 設計回覆金額 / 廠商歸屬 / 成本三合一規則（2026-04-03 夜間新增）
本段為今日最重要的新規則之一，用來正式接通設計線、文件主線與報價成本模組。

正式拍板：

> **設計回覆的確認行為，是「文件成立 + 廠商歸屬成立 + 成本成立」的三合一判定點。**

#### 三合一確認效果
當一筆設計回覆滿足：
1. 已選定執行廠商
2. 已按儲存完成確認

則同時成立三件事：

##### 1. 文件成立
- 該筆回覆正式成為已確認回覆
- 可進入設計文件整理層
- 可成為設計文件內容來源

##### 2. 廠商歸屬成立
- 該筆回覆正式歸屬到被選定的執行廠商名下
- 後續文件整理、廠商分組、成本歸戶都以此為準

##### 3. 成本成立
- 該筆回覆金額正式成立為成本
- 納入專案成本加總
- 可供 `報價成本` 模組承接
- 可供 `廠商資料` 模組承接到該廠商名下

#### 未確認 vs 已確認的成本差異
- **未確認回覆**
  - 不進文件整理層
  - 不進設計文件
  - 不歸屬到任何廠商成本
  - 不計入專案成本加總
- **已確認回覆**
  - 進文件整理層
  - 進設計文件
  - 歸屬到選定廠商
  - 計入專案成本加總

#### 修改規則（正式拍板）
若一筆已確認設計回覆後續被修改，則：

> **必須同步退出確認狀態、退出文件主線、退出成本主線。**

也就是：
- 原本已進文件整理層的資格取消
- 原本已計入的成本先撤回
- 原本已歸屬的廠商成本先失效
- 必須重新確認後，才重新成立

### E. 設計文件（2026-04-03 夜間定稿版）
文件型態：
- 表格型文件

正式定位：

> **依執行廠商整理後的正式設計輸出頁**

主次規則（正式拍板）：

> **以表格文件本體為主視覺核心，狀態與操作區為輔。**

#### 頁首識別區
保留：
- 執行廠商名稱
- 文件狀態
- 返回整理層入口

次資訊：
- 專案名稱
- 活動日期
- 地點

頁首規則：
- 先讓人知道這是哪個執行廠商的文件
- 再知道這份文件是不是最新
- 專案資訊只作背景資訊，不可喧賓奪主

#### 文件主表區
表格欄位：
- 編號
- 項目
- 尺寸
- 材質 + 結構
- 檔案位置（URL）
- 數量

資料來源規則（更新後）：
- **只有已確認回覆** 對應表格中的一列
- 編號由文件生成層自動產生

明確不進文件主表的欄位：
- 金額
- 負責人
- 狀態
- 需求說明
- 參考連結

#### 狀態與操作區
保留：
- 文件狀態說明
- `生成文件` / `重新生成文件`
- 次操作（如複製 / 匯出，若保留）

正式規則：
- 文件主表是主體
- 狀態與操作區是輔助，不可搶過主表
- `已生成` 時不顯示多餘主 CTA

#### 文件狀態規則
沿用統一狀態語言：
- `未生成`
- `已生成`
- `需更新`

##### `未生成`
- 代表已有已確認內容，但尚未生成正式設計文件
- 顯示：`生成文件`

##### `已生成`
- 代表目前這份文件就是最新版本
- 不顯示主 CTA，只保留狀態說明與次操作

##### `需更新`
- 代表已確認內容有變更，現有文件已不是最新
- 顯示：`重新生成文件`

#### 不該出現的錯誤
- 文件頁看起來像任務頁
- 表格不是主角
- 金額跑進表格
- 頁首資訊太多
- 狀態與操作搶過文件本體
- 使用者分不清現在這份是不是最新

# 17. 報價成本模組（2026-04-03 凌晨定稿版）

本節為 `projectflow` 新增的高優先主線之一。若後續討論報價單、成本、毛利、對帳、結案或廠商成本承接，原則上優先以本節為準。

## 17.1 模組定位
`報價成本` 是獨立模組，不掛在 `Project Detail` 內作為附屬區塊。

定位為：

> **以 project 為主體的報價、成本、對帳與結案管理模組**

核心目的：
1. 專案成本追蹤
2. 對外報價管理
3. 結案對帳

列表需分成：
- 執行中
- 已結案

## 17.2 收入端：對外報價單
### 對外報價單建立方式
- 以 **Excel 匯入** 建立
- 系統需保留每列明細
- 系統自動加總總金額

### 對外報價單的產品語意
- 對外報價單是 **收入端 / 對客金額基準**
- 對外報價總額不直接等於成本總額
- 後續毛利、差額、對帳都建立在這個收入基準之上

## 17.3 成本來源與成立門檻
### 成本來源
三條線的金額都會流進成本：
- 設計回覆金額
- 備品回覆金額
- 廠商報價

### 成本成立門檻（已定稿）
- 設計：進 `設計文件整理` 就算成本
- 備品：進 `備品整理` 就算成本
- Vendor：進 `廠商發包清單` 就算成本

重要原則：
- 不是等最終文件生成才算
- 只要進入整理主線 / package 主線，就視為正式成本明細

## 17.4 成本累計規則
已拍板：
- `R1 / R2 / R3` 若代表不同成本項，皆列入成本
- 不採版本覆蓋式只取單筆的邏輯
- 原始總成本 = 所有已進整理主線的成本明細加總

這代表成本模型是：

> **累加型成本模型，而不是覆蓋型成本模型**

## 17.5 關聯廠商規則
### 三條線統一規則
- 專案設計、專案備品、專案廠商，關聯廠商都應對到 **廠商資料庫**
- 匹配方式統一為：
  - 選單選取
  - 關鍵字匹配

### 關聯廠商的產品語意
關聯廠商不是自由文字，而是：

> **指向廠商資料庫實體的正式關聯欄位**

### 未匹配情況
若成本明細尚未成功匹配到廠商資料庫：
- 先進 `未指定廠商` 區
- 後續可再補綁正式 vendor

原則：
- 成本資料不能因尚未綁 vendor 而進不來
- 也不能用自由文字直接污染正式 vendor 主資料

## 17.6 成本管理：雙層模型
### A. 原始成本層
來源：
- 設計文件整理
- 備品整理
- 廠商發包清單

特性：
- 反映原流程中的原始成本來源
- 不因財務整理而被回改

### B. 成本管理層
位置：
- `報價成本` 模組內

特性：
- 可人工直接調整成本數字
- 作為最終對帳依據
- 不回寫原流程資料
- 不要求填寫調整說明

## 17.7 成本顯示規則
已拍板：
- 預設顯示 **調整後成本**
- 必要時可展開查看 **原始成本**

這條規則同時適用於：
- 專案總成本層級
- 廠商成本小計層級
- 單筆成本明細層級

## 17.8 成本管理區的主閱讀方式
### 以廠商為主分組
成本管理區的主軸不是先按：
- 設計
- 備品
- Vendor

而是先按：
- 廠商 A
- 廠商 B
- 廠商 C
- 未指定廠商

也就是：

> **專案中的成本，先按廠商聚合，再看廠商底下的明細與總額**

### 每個廠商區塊首屏
顯示：

### 廠商區塊展開後

### 每筆成本明細首屏
只需顯示：

### 每筆成本明細可做的操作

- 回看來源紀錄

## 18.2 模組之間的關係
### 主體關係
整體正確心智是：

> **Project Detail 是主控台，其他模組是沿著專案流程、財務流程、廠商流程往外長出的專責模組。**

### 連動關係
#### Project Detail → 專案設計 / 專案備品 / Vendor Flow
- 任務從 Project Detail 發出
- 往設計 / 備品 / 廠商三條線流動

#### 設計 / 備品 / Vendor Flow → 報價成本
- 三條線的成本來源進入 `報價成本`
- 其中 Vendor 與設計 / 備品的關聯廠商也會影響後續 vendor 模組承接

#### 報價成本 → 結案
- 執行中專案在 `報價成本` 做整理、對帳
- 完成後轉入 `結案`

#### 報價成本 / 專案成本 → 廠商資料
- vendor detail 會承接：
  - 該廠商在各專案中的調整後成本
  - 未付款狀態
  - 發包內容摘要與明細

#### 廠商資料 ↔ 設計 / 備品 / Vendor 流程
- 廠商主檔可被三條線匹配 / 選用
- 流程中找不到廠商時，也可 quick create vendor 並回填選中

- `尚無回覆`：該任務目前沒有任何設計回覆
- `待確認`：已有回覆，但目前尚無有效已確認回覆進文件主線，或仍有關鍵回覆尚待確認
- `已確認`：已有至少一筆已確認回覆，已具備進文件主線資格

#### 第二層文件狀態

- `未生成`
- `已生成`
- `需更新`

- 條件型次操作：`查看整理內容` / `查看文件`
- 首頁不直接強推 `生成文件` 作為主 CTA

狀態對應：
- `尚無回覆`：主操作 `查看任務`，不顯示整理 / 文件導流
- `待確認`：主操作 `查看任務`，仍以回覆確認主場為主
- `已確認`：主操作 `查看任務`，次操作可出現 `查看整理內容`，若文件已存在或有文件狀態時再顯示 `查看文件`
