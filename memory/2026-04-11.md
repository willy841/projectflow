- Root cause of unstable design summary CTA was a **double-source state problem** around `boardPath`:
  - `ExecutionTree` saved assignment state only stored draft data
  - `ExecutionTreeSection` separately patched `boardPath` into summary state
  - later syncs from saved assignments could wipe `boardPath`, causing CTA to fall back from detail link to board link
- Fix applied for `boardPath` stability:
  - `savedDesignAssignments`, `savedProcurementAssignments`, `savedVendorAssignments` in `src/components/execution-tree.tsx` were changed to store `{ data, boardPath }`
  - outbound assignment sync now includes `boardPath`
  - this stabilized reload behavior so summary CTA remained `前往設計任務詳情` with persistent `/design-tasks/<uuid>` href
- A direct field-mapping issue was confirmed for design dispatch:
  - DB inspection showed `design_tasks.structure` had been receiving note-like content (`前端驗收路線`) instead of proper structure data
  - immediate stopgap change in `src/components/execution-tree-section.tsx` changed the dispatched `structure` payload from `draft.structureRequired` to `draft.material` to stop writing note-like values into `structure`
  - however, this is only a stopgap; the durable unresolved issue is that the current design form uses a merged UI field `材質 + 結構` while downstream DB/detail/document still expect separate `material` and `structure` semantics
