## 2. List 頁：資料來源與欄位定義

### 使用情境
- 進入 `報價成本` 後，先看到以 `專案` 為主體的 list 頁面
- 資料來源應承接上游 project：建立專案後，報價成本這邊就要同步顯示該專案存在

### List 欄位
1. `標題`
   - 顯示專案名稱
2. `專案活動時間`
3. `狀態`
   - 用來確認報價單是否上傳
   - 若尚未上傳，顯示 `未上傳`
   - 上傳後，顯示 `已上傳`

### 補充規則
- 目前前端 UI 的 `未結案` badge 可以拿掉
- 因為只要專案結案了，它就不會出現在這個 list 頁面裡
- 結案的案子應到 `結案紀錄`

### 判斷
- 功能邏輯：正確
- 責任邊界：正確
- `報價成本 list` 是 project 的 financial entry list，不應自己養另一套 project source

### 目前狀態
- `建立專案 -> 同步出現在報價成本 list`：大方向吻合現有正式架構
- `未結案` badge：屬冗餘資訊，應列為待移除 UI 項目
- `已上傳 / 未上傳` 狀態欄：產品定義正確，但後續仍需驗證是否 truly same-source 承接 quotation 主線

---

## 3. Detail 頁上方：專案資訊與狀態提醒

### 使用情境
進入 detail 頁後，最上方區塊為 `專案資訊`：
- `標題`
- `日期`

### 欄位調整
1. `現在的客戶`
   - 可拿掉
2. `對帳狀態` / `結案狀態`
   - 可保留
   - 但這裡只是狀態提醒，不需要 `返回報價成本列表` 按鈕

### 判斷
- 功能邏輯：正確
- 責任邊界：正確
- 最上方區塊應聚焦 project identity + status reminder，而不是導覽 CTA 區

### 目前狀態
- `現在的客戶`：應列為可移除欄位
- `返回報價成本列表`：若仍存在於狀態區，應列為待移除 UI 項目

## 4. 四張 financial summary 卡

### 使用情境與來源定義
1. `應收總金額`
   - 來自對外報價單匯入後抓到的最後總額
2. `已收款`
   - 對應下方 `收款管理`
   - 透過 `新增收款` 後顯示在這裡
3. `未收款`
   - `應收總金額 - 已收款`
4. `毛利`
   - `應收總金額 - 所有開銷`
   - 開銷包含：
     - 設計
     - 備品
     - 廠商
     - 人工費用

### 判斷
- 四張卡的功能邏輯正確
- 與 financial detail 的正式架構吻合
- 與前面三條執行線（設計 / 備品 / 廠商）的 SoT 承接方向一致

### 後續值得精驗
1. `應收總金額` 是否 truly 承接 quotation 正式主線
2. `已收款` 是否 truly same-source 對應收款管理
3. `毛利` 是否 truly 承接全部正式成本 bucket，而不是部分 fallback / transitional 值

## 7. 成本管理（四個 Tab / 四個成本 bucket）

### 使用情境
成本管理包含四個來源：
1. `設計`
2. `備品`
3. `廠商`
4. `人工`

### SoT 規則
- `設計 / 備品 / 廠商`
  - 來源不是在這裡手填成本主真值
  - 它們的 Source of Truth 來自各自執行線，在 `全部確認` 後正式帶入
- `人工`
  - 不是來自前面三條執行線
  - 是在這裡直接新增人工成本
  - 按下 `儲存` 後成立

### 判斷
- 功能邏輯：正確
- 與前面三條執行線完全吻合
- 人工作為 financial detail 內部自行維護的第四個成本 bucket，也是合理的正式架構

### 後續值得精驗
1. 設計 / 備品 / 廠商成本是否 truly same-source 從三條執行線帶入
2. 人工成本的 `儲存` 是否確實等同正式成立
3. 卡片 / Tab / 明細之間是否 truly same-source

---

## 8. 對帳管理（對帳群組）

### 使用情境
- 對帳管理以 `廠商` 為主體做 `對帳群組`
- 因為在前面的設計 / 備品 / 廠商階段，執行回覆時都已選擇了廠商
- 所以在這裡以 vendor 為主體對帳是合理的

### 對帳顯示要求
- 不能只顯示一個群組總額
- 必須看到該廠商底下帶了哪些任務、哪些金額
- 才能確認金額項目是否正確

### 已定規則
- 對帳後的金額，在按下 `已對帳` 以後，會成為 `廠商資料` 中個別廠商 `未付款金額` 的 Source of Truth
- 廠商資料裡的金額是各筆已對帳資料的 `加總累積`
- 來源只包含：
  - `設計`
  - `備品`
  - `廠商`
- **不包含 `人工`**

### 判斷
- 功能邏輯：正確
- 與 vendor 主線 / payable 主線的正式架構吻合
- `已對帳` 的責任是讓 vendor unpaid 正式成立，而不是重新決定原始成本真值

### 後續值得精驗
1. 對帳群組是否真的能展開看到任務與金額明細，而不只是總額
2. `已對帳` 後 vendor 資料中的未付款金額，是否 truly same-source 來自這裡
3. 累積邏輯是否正確區分：
   - vendor payable = 累積
   - execution-line confirm-all = 覆蓋

## 9. 結案動作與結案條件

### 使用情境
- `報價成本` detail 同時是 `結案` 動作的管理位置
- 系統必須在滿足條件後，才能執行結案

### 結案條件
必須同時滿足：
1. `未收款 = 0`
   - 代表應收款項都已收完
2. `款項全部對帳完畢`
   - 代表 vendor 對帳都已完成

### 正式規則
- 只有在上述兩個條件都成立後，系統才可執行 `結案`
- 在未滿足條件之前，不可允許按下 `結案` 按鈕

### 目前 UI 缺口
- 目前 UI 上尚未提供 `結案` 按鈕
- 因此應補上此正式入口
- 並加上條件 gating：未滿足條件前，按鈕不得可用

### 判斷
- 功能邏輯：正確
- 與報價成本線的 financial closure 責任吻合
- 與 `未收款` / `對帳管理` 兩條既有主線完全一致

### 後續值得精驗
1. `未收款 = 0` 是否 truly 由正式 financial source 判斷
2. `全部對帳完畢` 是否有明確且正式的狀態判斷來源
3. 結案後是否 truly 會自報價成本 list 移出並進入結案紀錄
4. 結案按鈕是否在條件未滿足前被正確 gating

3. 結案案子應分流到結案紀錄
4. Detail 上方聚焦標題 / 日期 / 狀態提醒
5. 四張 summary 卡的 financial 邏輯成立

5. 四張卡 / 成本明細 / 對帳群組之間是否 truly same-source
6. `最終文件內容` 區塊應移除

8. `結案` 按鈕目前缺失，需補上且正確 gating

## 12. 一句話總結

> 目前已核對的 `projectflow` 報價成本線使用情境整體是清楚且與正式架構方向吻合的：List 以 project 為主體承接上游專案，detail 上方聚焦 project identity 與狀態提醒，四張 financial summary 卡承接應收 / 已收 / 未收 / 毛利，對外報價單透過 Excel 匯入且總額直接抓 Excel 欄位，收款管理回寫已收款 / 未收款，成本管理承接設計 / 備品 / 廠商三條執行線在 `全部確認` 後成立的成本 SoT，人工則在此直接新增；而最重要的對帳管理則應以 vendor 為主體、展開任務與金額明細後再按 `已對帳`，使 vendor 資料中的未付款金額正式成立且累積加總（不含人工）。此外，`結案` 應由報價成本 detail 管理，且必須以 `未收款 = 0` 與 `全部對帳完畢` 作為前置條件 gating。後續最值得處理的是 UI 冗餘移除、same-source 驗證、結案按鈕補齊，以及對外報價單 Excel 欄位 mapping spec 的補定稿。
